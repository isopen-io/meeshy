## Constat — 4 champs credential sans `textContentType` (AutoFill inerte)

Le flux 2FA expose **4 saisies de credential**, toutes dépourvues de
`textContentType` → iOS ne peut proposer **aucune suggestion AutoFill** :

1. `codeEntryView` (setup) — `TextField` code 6 chiffres (`verificationCode`)
2. `TwoFactorDisableView` — `SecureField` mot de passe (`disablePassword`)
3. `TwoFactorDisableView` — `TextField` code 6 chiffres (`disableCode`)
4. `TwoFactorBackupCodesView.codeEntryStep` — `TextField` code 6 chiffres

Les 3 champs code avaient déjà `.keyboardType(.numberPad)` mais **pas**
`.textContentType(.oneTimeCode)` — le content-type sémantique qu'Apple
recommande pour tout code de vérification à usage unique. Sans lui, la bannière
« codes de vérification » (Messages / app Mots de passe iOS 17+ pour un TOTP
stocké) au-dessus du clavier n'apparaît jamais. Le `SecureField` mot de passe
n'avait pas `.textContentType(.password)` → aucune proposition Trousseau.

Résultat : l'utilisateur re-saisit à la main un code/mot de passe qu'iOS aurait
pu remplir en un tap. Friction évitable, contraire à l'objectif « reduce number
of taps / native platform integration ».

## Correctif (208i)

- `.textContentType(.oneTimeCode)` ajouté après `.keyboardType(.numberPad)` sur
  les **3** `TextField` de code (setup / disable / backup-codes).
- `.textContentType(.password)` ajouté sur le `SecureField` mot de passe du flux
  de désactivation → AutoFill Trousseau du mot de passe de compte existant.

`.oneTimeCode` est le content-type Apple-canonique pour un code de vérification
à usage unique et cohabite avec `.numberPad` (iOS pose la bannière AutoFill
au-dessus du pavé numérique). `.password` sur un champ de re-saisie du mot de
passe de compte est exactement le cas d'usage Trousseau attendu.
