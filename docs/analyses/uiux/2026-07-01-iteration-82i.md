# Analyse UI/UX — Itération 82i (iOS)

**Date** : 2026-07-01
**Piste** : iOS (suffixe `i`) — indépendante des pistes web/android
**Surface** : `apps/ios/Meeshy/Features/Main/Views/ActiveSessionsView.swift`
**Type** : Accessibilité — Dynamic Type (polices figées → styles relatifs scalables)

## Contexte

Écran de sécurité « Sessions actives » (liste des sessions/appareils connectés,
révocation individuelle et globale). Contenu 100 % **texte de lecture** : nom
d'appareil, adresse IP, horodatage de dernière activité, badge « Actuelle »,
titre, boutons. C'est exactement le type de surface où le respect du réglage
Dynamic Type de l'utilisateur est critique (HIG Accessibility : « Use semantic
fonts, not fixed sizes for body text »).

Avant l'itération, **12** call sites utilisaient `.font(.system(size:))` figé —
donc **aucun** ne scalait avec le réglage « Larger Text » de l'utilisateur.

## Diagnostic (état du métier)

- **Apple HIG / iOS Accessibility** : les textes de lecture doivent adopter des
  styles typographiques relatifs (`.body`, `.caption`, etc.) pour scaler avec
  Dynamic Type. Un `.system(size: 14)` figé casse cette promesse.
- **SSOT projet** : le helper `MeeshyFont.relative(size, weight:, design:)`
  (`packages/MeeshySDK/Sources/MeeshyUI/Theme/Accessibility.swift:152`) mappe une
  taille en points legacy vers le `Font.TextStyle` le plus proche tout en
  préservant weight/design. C'est le pattern de migration retenu et déjà mergé
  en 74i (`ConversationDashboardView`), 71i/72i (2FA, feed comments), etc.
- Migration = swap mécanique 1:1, aucune logique touchée.

## Périmètre délibérément exclu (ne pas re-flagger)

- **Ligne 109** — icône d'appareil (`iphone`/`desktopcomputer`) `.system(size: 16,
  .medium)` **conservée figée** : elle est centrée dans un badge à **géométrie
  fixe 32×32** (`.frame(width: 32, height: 32)` + fond `RoundedRectangle`). La
  faire scaler la ferait déborder du conteneur aux tailles AX. Décision cohérente
  avec le précédent 74i (« glyphs inside fixed rings stay fixed »).
