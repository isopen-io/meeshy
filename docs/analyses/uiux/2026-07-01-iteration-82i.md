# Analyse UI/UX — Iteration 82i (2026-07-01) — Dynamic Type `AudioFullscreenView`

## Contexte
Piste iOS (suffixe `i`), indépendante des itérations web. Poursuite de la campagne Dynamic Type
sur les grandes surfaces immersives restantes, listée comme différé prioritaire iOS depuis 71i
(« AudioFullscreenView (26) »). Surface **libre** (aucune PR ouverte au moment du run — forte
contention iOS, ~10 agents ; vérifié via `list_pull_requests`).

## Surface
`apps/ios/Meeshy/Features/Main/Views/AudioFullscreenView.swift` — lecteur audio plein écran
(pages swipe, waveform seek, contrôles transport, sélecteur de langue Prisme Linguistique,
transcription). 900 lignes, 26 `.font(.system(size:))` figées, immersif fond sombre.

## Problème
Les tailles de police figées (`.system(size:)`) ne réagissent PAS au réglage Dynamic Type
(Réglages > Accessibilité > Texte plus grand). Un utilisateur malvoyant qui agrandit le texte
système voit le lecteur audio inchangé — noms d'expéditeur, horodatage, temps de lecture,
labels de langue, texte de transcription restent en corps fixe. Régression d'accessibilité
(WCAG 1.4.4 Resize Text) sur une surface média centrale.

## Doctrine appliquée (identique 55i / 71i, SSOT `MeeshyFont.relative`)
`MeeshyFont.relative(size, weight:, design:)` (`MeeshyUI/Theme/Accessibility.swift`) mappe une
taille en points vers le `Font.TextStyle` relatif le plus proche → scaling Dynamic Type, en
préservant weight et design (dont `.monospaced`). Swap purement mécanique.

Distinction retenue :
- **Texte + glyphe de contenu inline libre** (dans un conteneur extensible) → scale.
- **Glyphe de chrome/transport contraint par `.frame(width:height:)` fixe** → figé (le glyphe
  agrandi déborderait le cercle du contrôle → clipping ; les transports média ne scalent pas, HIG).
- **Icône décorative héros ≥ 28 pt** → figée (hiérarchie visuelle empty-state).
