# UI/UX Analysis — Iteration 93i (2026-07-01)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android couverts par d'autres agents).
Thème : **accessibilité — Dynamic Type + VoiceOver** sur `LocationPickerView` (composant de
sélection de lieu : carte MapKit interactive, barre de recherche flottante, liste de résultats,
carte d'action inférieure « Ma position » / « Confirmer »). Migration mécanique des polices figées
`.font(.system(size:))` vers `MeeshyFont.relative(...)` (scaling Dynamic Type) + masquage VoiceOver
des glyphes décoratifs appariés à du texte.

Vérification : la CI `iOS Tests` (`ios-tests.yml`, compile Xcode 26.1.x + tests simulateur 18.2)
sert de build de validation (SwiftUI ne compile pas sous Linux). **Aucun test neuf** : sweep
typographique pur + 4 traits a11y déclaratifs, aucune logique modifiée (parité 55i / 74i / 86i /
88i / 90i). `MeeshyFont` est déjà en scope (`import MeeshyUI` ligne 5).

## Contexte / point de départ
`LocationPickerView.swift` (391 lignes) est le picker de localisation ouvert depuis le composer
(pièce jointe « lieu ») : `NavigationStack` + `ZStack` (carte plein écran + barre de recherche en
haut + carte d'action en bas). La palette est **déjà** entièrement tokenisée : accent déterministe
`Color(hex: accentColor)` (teinte de conversation), `theme.textPrimary/textSecondary/textMuted`,
et surfaces `.adaptiveGlass(...)` (Liquid Glass iOS 26 neutre + fallback `.ultraThinMaterial`, déjà
gated par le wrapper SDK Compatibility). Aucune dette palette. Mais **17 sites** de police fixaient
une taille absolue `.font(.system(size:))` — l'écran ignorait entièrement le réglage Dynamic Type
(rupture directe de la règle a11y CLAUDE.md « Use semantic fonts, NEVER use fixed font sizes for
body text »). Cet écran est un flux de saisie utilisateur (adresse, coordonnées) qui doit rester
lisible à toutes les tailles.

`Font.Design`, signature `Accessibility.swift:152`).

### VoiceOver — 4 traits déclaratifs
- `.accessibilityAddTraits(.isHeader)` sur le titre toolbar (rotor « En-têtes », convention 86i).
- `.accessibilityHidden(true)` sur 3 glyphes décoratifs appariés à du texte : loupe de recherche
  (le placeholder du champ porte le sens), `mappin` dans son badge fixe (le nom du résultat porte
  le sens), `location.fill` de la carte d'action (l'adresse adjacente porte le sens). VoiceOver ne
  prononce plus les noms de symbole SF redondants.

## Périmètre délibérément exclu (ne pas re-flagger)

### `mappin.circle.fill` marqueur de carte (ligne 70) — 36pt gardé figé
Glyphe **annotation de carte** ancré à une coordonnée MapKit, rendu à une taille d'écran fixe
comme une épingle système. Le scaler avec Dynamic Type le détacherait visuellement du point qu'il
marque. **Décision : conservé `.font(.system(size: 36))`** + commentaire d'exception (doctrine
74i/86i — ornement/chrome de grande taille figé).

### `mappin` glyphe de résultat (ligne 144) — 12pt dans badge fixe 28×28
Glyphe contraint dans un `.frame(width: 28, height: 28)` + `Circle` de fond. Une police scalable
déborderait/tronquerait le glyphe hors du badge fixe → **doctrine 86i** (`fieldIcon` d'`AboutView`,
`toggleRow` de `DataExportView` 90i : glyphes contraints dans un conteneur fixe restent figés).
**Conservé `.font(.system(size: 12, weight: .semibold))`** + `.accessibilityHidden(true)` +
commentaire in-situ.

### Couleurs — déjà entièrement tokenisées (rien à faire)
Accent déterministe `Color(hex: accentColor)`, `theme.textPrimary/textSecondary/textMuted`,
surfaces `.adaptiveGlass` neutres (chrome par choix, pas de tint accent). Aucune couleur hardcodée
hors-charte. **Aucun swap palette dans ce lot.**

### i18n — déjà couvert
Toutes les chaînes sont localisées via `String(localized:defaultValue:bundle:.main)` (clés
`location.*` + `common.*`). Le `String(format: "%.5f, %.5f", …)` des coordonnées est une donnée
numérique brute (non traduisible). **0 clé neuve.**

### Glass / gestes / navigation — conformes
`.adaptiveGlass` déjà en place (Liquid Glass iOS 26 gated SDK), swipe-to-back natif du
`NavigationStack`, `dismiss()` via bouton toolbar `.cancellationAction` + `HapticFeedback.success()`
à la confirmation. Rien à modifier.
