# UI/UX Analysis — Iteration 71i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **Dynamic Type sur le flow sécurité 2FA** (`TwoFactorSetupView.swift`) — migration
mécanique des tailles de police figées `.font(.system(size:))` vers le helper scalable
`MeeshyFont.relative(...)`. Itération bornée à **un seul fichier de production**, dans la
continuité directe du différé « Dynamic Type grandes surfaces restantes » listé par 55i
(`TwoFactorSetupView` 39 occ.) et 70i.

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de
build de validation (SwiftUI ne compile pas sous Linux). Le swap est purement typographique,
sans changement de logique ni de modèle — aucun nouveau test requis (le helper
`MeeshyFont.relative` / `textStyle(for:)` est déjà couvert par `AccessibilityHelpersTests`).

## Contexte / point de départ
- `TwoFactorSetupView.swift` regroupe **trois écrans** du flow 2FA : `TwoFactorSetupView`
  (QR + clé + saisie code + codes de secours), `TwoFactorDisableView` (désactivation),
  `TwoFactorBackupCodesView` (régénération des codes de secours).
- L'écran était truffé de `.font(.system(size: N, weight:))` **figés** : instructions, titres,
  boutons CTA, messages d'erreur, libellés de copie, champs de saisie du code à 6 chiffres.
  Ces tailles ne réagissaient **pas** au réglage Dynamic Type du système → un utilisateur
  malvoyant ayant augmenté la taille du texte voyait le flow de sécurité rester en petit corps.
- Le helper `MeeshyFont.relative(_:weight:design:)` (SDK `MeeshyUI/Theme/Accessibility.swift`,
  introduit par les itérations 55i/68i) mappe une taille legacy vers le `Font.TextStyle`
  relatif le plus proche, préservant `weight` et `design` — c'est le swap canonique établi.

## iOS Findings

### Migration Dynamic Type — `.system(size:)` → `MeeshyFont.relative`
**28 sites** de texte ou d'icône inline convertis (tout le corps lisible du flow) :
- Instructions multilignes (scan QR, saisie code, avertissements, codes de secours).
- Titres de succès / désactivation (« 2FA activé », « Désactiver l'authentification… »).
- Tous les CTA capsule (« Suivant », « Vérifier et activer », « Terminer »,
  « Confirmer la désactivation », « Générer les codes »).
- Messages d'erreur de code (`codeError`, `disableError`).
- Libellés de copie + icônes inline `doc.on.doc` / `doc.on.doc.fill` (scalent avec leur
  texte voisin).
- Champs de saisie du code à 6 chiffres (`TextField`/`SecureField`, monospace 28 → `.title`)
  et mot de passe — désormais lisibles à grande taille.

Le design `monospaced` et chaque `weight` sont préservés à l'identique → rendu visuel inchangé
au réglage Dynamic Type par défaut, et scaling correct au-delà.

### Périmètre délibérément exclu (ne pas re-flagger)
- **9 glyphes SF Symbol illustratifs hero** gardés figés (`qrcode` 80, `lock.shield.fill` 50,
  `checkmark.shield.fill` 50, `key.fill` 40, `shield.slash.fill` 50,
  `exclamationmark.triangle(.fill)` 40/50 ×4) : décoratifs et autonomes au-dessus d'un texte
  qui, lui, scale et porte le sens. Les faire grossir jusqu'à AX5 ferait exploser la hauteur
  des écrans de scroll (le QR lui-même est en `frame(width: 200, height: 200)` fixe). Conforme
  à la doctrine ContactCardView 70i (« grandes illustrations figées »).
- **2 grilles de codes de secours** (`LazyVGrid` 2 colonnes, monospace 14) gardées figées +
  **commentées** : un code de secours est sécurité-critique ; à grande taille dans une cellule
  bornée 2 colonnes il risquerait la troncature/débordement. Conforme à la doctrine « layout
  borné → figé ».
- `setup.otpauthUrl` est déjà en text-style relatif (`.system(.subheadline, design:
  .monospaced)`) avec `minimumScaleFactor(0.7)` → déjà scalable, laissé tel quel.

## Conforme (vérifié — ne pas re-flagger)
- Le secret OTP et les codes de secours sont déjà effacés du presse-papiers après 30 s
  (sécurité presse-papiers HIG). Conforme.
- Boutons de copie déjà dotés de `.accessibilityLabel` localisés. Conforme.
- Couleurs : `tfaColor = MeeshyColors.indigo500` + tokens sémantiques (`success`/`error`/
  `warning`) — aucune couleur hex dure. Conforme.

antérieure n'avait touché `TwoFactorSetupView`. Le différé Dynamic Type 55i+ le listait
explicitement (« `TwoFactorSetupView` (39) ») sans l'avoir traité.
