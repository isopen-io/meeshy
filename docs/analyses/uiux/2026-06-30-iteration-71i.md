# UI/UX Analysis — Iteration 71i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **Dynamic Type / accessibilité (4y11) du flux de sécurité 2FA** —
`TwoFactorSetupView.swift`, qui héberge **trois** écrans (`TwoFactorSetupView`,
`TwoFactorDisableView`, `TwoFactorBackupCodesView`). Itération bornée à **un seul fichier de
production**, dans la continuité directe du différé « Dynamic Type grandes surfaces » listé par
55i/69i/70i (`TwoFactorSetupView` = 39 tailles de police figées).

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de
build de validation (SwiftUI ne compile pas sous Linux). Ce lot est un **sweep typographique
mécanique** sans logique pure isolable — pas de nouveau test unitaire (parité avec 55i
`ConversationInfoSheet`, autre sweep Dynamic Type pur) ; le gate est la compile CI + la suite
existante.

## Contexte / point de départ
- Le flux 2FA (`apps/ios/Meeshy/Features/Main/Views/TwoFactorSetupView.swift`) — configuration
  TOTP, désactivation, régénération de codes de secours — figeait **39** tailles de police via
  `.font(.system(size:))`. Tout texte (instructions, titres, champs de saisie code, codes de
  secours monospacés, libellés de boutons, messages d'erreur) restait à taille **fixe** quelle
  que soit la préférence Dynamic Type de l'utilisateur.
- Conséquence a11y : un utilisateur ayant agrandi la police système (Réglages › Accessibilité ›
  Texte plus grand) voyait un écran de **sécurité critique** au texte non agrandi — le code à
  6 chiffres à saisir et les codes de secours à mémoriser restaient minuscules. C'est l'un des
  écrans où la lisibilité importe le plus (saisie sécurisée sous pression).

## iOS Findings

### Dynamic Type — `.system(size:)` figé → `MeeshyFont.relative(...)` scalable (texte)
**30 conversions** sur les call sites de **texte** et **petites icônes inline** :
- Tous les `Text` / `TextField` / `SecureField` : instructions, titres (« 2FA activé ! »,
  « Désactiver… »), champs de saisie du code (`relative(28, weight: .bold, design: .monospaced)`),
  codes de secours en grille (`relative(14, weight: .semibold, design: .monospaced)`), libellés
  de boutons (Suivant / Vérifier / Terminer / Confirmer / Générer), messages d'erreur, libellés
  « Copier ». Le swap est **mécanique** et préserve **weight + design** (cf. signature
  `MeeshyFont.relative(_:weight:design:)`, `packages/MeeshySDK/Sources/MeeshyUI/Theme/Accessibility.swift`).
- **Petites icônes inline `doc.on.doc`/`checkmark` (12–13 pt)** accolées à un libellé : converties
  aussi, pour qu'elles **suivent** la taille du texte voisin (parité 55i qui convertissait les
  glyphes 10 pt).
- Gain : le texte du flux 2FA scale désormais avec la préférence Dynamic Type, sur les **trois**
  écrans (setup, disable, backup-codes).

### Périmètre délibérément exclu (ne pas re-flagger)
- **9 icônes héros décoratives `Image(systemName:)` 40/50/80 pt gardées figées**
  (`qrcode` 80, `lock.shield.fill`/`checkmark.shield.fill`/`shield.slash.fill` 50,
  `exclamationmark.triangle`/`key.fill` 40). Les replier toutes vers `MeeshyFont.relative`
  les **écraserait toutes sur `.largeTitle`** (`textStyle(for:)` mappe 40/50/80 → largeTitle) →
  **perte de la hiérarchie visuelle** (le QR 80 pt deviendrait égal aux boucliers 50 pt). Les
  icônes héros décoratives n'ont pas vocation à scaler (HIG) ; elles restent au point fixe.
- `setup.otpauthUrl` (ligne 115) : **déjà** Dynamic Type via `.font(.system(.subheadline,
  design: .monospaced)...)` (style relatif, pas `size:`). Non touché.

## Conforme (vérifié — ne pas re-flagger)
- Cibles tactiles : tous les boutons ont un padding vertical ≥ 12 → ≥ 44 pt effectifs. Conforme.
- Sécurité presse-papier : copie clé/codes effacée après 30 s (déjà en place). Conforme.
- Couleurs : `tfaColor = indigo500`, états via `MeeshyColors.success`/`.error`/`.warning` —
  tokens charte, pas de hex dur. Conforme.
- a11y labels présents sur les boutons « Copier ». Conforme.
