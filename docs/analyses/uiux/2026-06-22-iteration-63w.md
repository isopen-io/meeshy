## Problème traité — couleurs codées en dur sur les empty states (rupture dark/light)
Le reste du dossier `components/notifications/` (NotificationItem, NotificationFilters,
NotificationList, PushPermissionBanner) utilise **systématiquement** les design tokens
HSL (`bg-card`, `bg-muted`, `text-muted-foreground`, `text-foreground`, `border-border`).
Deux composants **empty state rendus en production** étaient les seuls **hors charte**,
codant des couleurs Tailwind brutes `gray-*` / `white` :

1. **`components/notifications/NotificationEmptyState.tsx`** (rendu par
   `NotificationList` + `NotificationDropdown`) — conteneur `bg-white/60 dark:bg-gray-900/60`,
   bordure `border-white/30 dark:border-gray-700/40`, cercle d'icône
   `from-gray-200 to-gray-300 dark:from-gray-700 dark:to-gray-800`, icône
   `text-gray-400 dark:text-gray-500`, titre `text-gray-900 dark:text-white`,
   description `text-gray-600 dark:text-gray-400`.
2. **`components/conversations/ConversationEmptyState.tsx`** — conteneur
   `bg-white/50 dark:bg-gray-950/50` ; **bouton primaire** avec `text-white` figé
   au lieu du token sémantique **`text-primary-foreground`** (le « sur-primaire »
   défini par la charte) — bug de contraste potentiel si `--primary-foreground`
   diffère du blanc dans un thème futur.

**Pourquoi c'est un bug Prisme/charte** : les couleurs codées en dur ne suivent pas
les variables CSS `--card`/`--muted`/`--border`/`--primary-foreground`, donc elles
**ne réagissent pas** à un changement de charte et peuvent rompre le contraste en
dark/light. Single Source of Truth des couleurs = tokens, jamais `gray-*`/`white`.

### Bug type latent corrigé en passant
`NotificationEmptyState` destructurait `_isSearching` alors que le type expose
`isSearching` → **TS2339** (la prop passée par les deux parents était silencieusement
perdue). Corrigé en `isSearching: _isSearching` (prop reconnue, binding marqué inutilisé
par convention `_`). Aucun changement de comportement (la prop n'altère pas le rendu).
