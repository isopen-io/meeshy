# UI/UX Analysis — Iteration 83i (2026-07-01)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android couverts par d'autres agents).
Thème : **accessibilité Dynamic Type + parité i18n sur les écrans légaux** — les deux vues
plein-texte `TermsOfServiceView` et `PrivacyPolicyView` figeaient toute leur typographie en
`.font(.system(size:))`, et `PrivacyPolicyView` avait encore une date « dernière mise à jour »
codée en dur en ternaire FR/EN (rupture du Prisme / i18n). Or ce sont précisément des écrans de
**lecture longue de texte juridique** où le Dynamic Type est le plus critique pour l'accessibilité
(un malvoyant doit pouvoir agrandir le corps du texte).

Vérification : la CI **iOS Tests** (compile Xcode 26.1.1 / Swift 6.2 + tests simulateur 18.2) sert
de build de validation (SwiftUI ne compile pas sous Linux). **Aucun test neuf** : sweep purement
typographique + swap i18n littéral→`String(localized:)`, aucune logique modifiée. `MeeshyFont` et
`Color(hex:)` sont déjà en scope (les 2 fichiers importent `MeeshyUI`).

## Contexte / point de départ
Doctrine Dynamic Type établie (55i / 71i / 72i / 74i) : migrer le **texte de lecture** et les
**glyphes inline accolés à un libellé** de `.font(.system(size:))` vers `MeeshyFont.relative(size,
weight:, design:)` (mappe la taille legacy vers le `Font.TextStyle` relatif le plus proche, en
préservant weight/design — cf. `MeeshyUI/Theme/Accessibility.swift`). Garder figés : chrome à frame
fixe, contrôles de précision, héros décoratifs ≥ 40 pt.

Les deux écrans légaux partagent une structure **identique** (header retour + titre, picker de
langue segmenté, ligne « dernière mise à jour », N cartes de sections numérotées). `TermsOfServiceView`
localisait déjà sa date (`legal.terms.lastUpdated.fr/.en`) ; `PrivacyPolicyView` ne l'avait jamais
fait — asymétrie corrigée ici.

## iOS Findings

### A. Dynamic Type — `.font(.system(size:))` → `MeeshyFont.relative(...)` (accessibilité)
**14 conversions** (7 par fichier, structures miroir) — 100 % du texte de lecture + glyphes inline :

| Rôle | Avant | Après |
|---|---|---|
| chevron retour (inline, accolé au libellé) | `.system(size: 14, weight: .semibold)` | `MeeshyFont.relative(14, weight: .semibold)` |
| libellé « Retour » | `.system(size: 15, weight: .medium)` | `MeeshyFont.relative(15, weight: .medium)` |
| titre d'écran (`.isHeader`) | `.system(size: 17, weight: .bold)` | `MeeshyFont.relative(17, weight: .bold)` |
| ligne « dernière mise à jour » | `.system(size: 12, weight: .medium)` | `MeeshyFont.relative(12, weight: .medium)` |
| numéro de section (glyphe inline `N.circle.fill`) | `.system(size: 18, weight: .semibold)` | `MeeshyFont.relative(18, weight: .semibold)` |
| titre de section | `.system(size: 15, weight: .bold)` | `MeeshyFont.relative(15, weight: .bold)` |
| **corps de section (texte juridique long)** | `.system(size: 14, weight: .regular)` | `MeeshyFont.relative(14, weight: .regular)` |

Fichiers : `TermsOfServiceView.swift`, `PrivacyPolicyView.swift`.

### B. i18n — date « dernière mise à jour » de `PrivacyPolicyView` (parité Prisme)
`PrivacyPolicyView.swift:132` : ternaire FR/EN codé en dur
`selectedLanguage == "fr" ? "Derniere mise a jour : 24 fevrier 2026" : "Last updated: February 24, 2026"`
→ `String(localized: "legal.privacy.lastUpdated.fr"/".en", defaultValue: …, bundle: .main)`.
Miroir exact du pattern déjà en place dans `TermsOfServiceView` (`legal.terms.lastUpdated.*`).
Pattern **defaultValue-only** (aucune entrée `.xcstrings` — vérifié : `legal.terms.lastUpdated`
n'existe que dans le `.swift`) → cohérence stricte avec Terms, zéro fichier catalogue à toucher.

### Gain
- **Accessibilité** : le corps du texte juridique (le contenu le plus important à pouvoir agrandir)
  scale désormais avec le réglage Dynamic Type de l'utilisateur — conformité WCAG 1.4.4 (resize text).
- **Parité Prisme/i18n** : la date de la politique de confidentialité passe par le catalogue de
  localisation comme le reste de l'écran (plus d'îlot FR/EN codé en dur).
- **Cohérence** : les deux écrans légaux, jumeaux structurels, sont désormais typographiquement et
  i18n-parfaitement symétriques.

## Conforme (vérifié — NE PAS re-flagger)
- **`accentColor` (`"45B7D1"` Terms / `"6366F1"` Privacy)** laissé en hex littéral : c'est un accent
  **décoratif per-écran** passé comme paramètre **hex-string** aux API `theme.surfaceGradient(tint:)`
  / `theme.border(tint:)` (qui attendent une `String`). Le tokeniser serait invasif (churn d'API) et
  hors doctrine « swap near-token seulement si exact + non-invasif » (78i). Même si `6366F1` = indigo500,
  l'usage comme tint-string ne justifie pas la conversion ici.
- **Frame chrome `Color.clear.frame(width: 60, height: 24)`** (spacer d'équilibrage du header) :
  volontairement fixe, `accessibilityHidden(true)`.

## Périmètre délibérément exclu (ne pas re-flagger)
- Contenu juridique lui-même (dictionnaires `sections`) : texte produit, hors-scope typo/i18n
  technique (déjà bilingue FR/EN via le picker in-view — design produit assumé).
- `accentColor` décoratif (voir ci-dessus).
