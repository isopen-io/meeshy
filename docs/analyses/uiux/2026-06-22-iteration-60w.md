## Problème traité — modal de paramètres entièrement FR figé
`components/settings/config-modal.tsx` (modal global de configuration, navigation
par onglets, monté depuis le header/menu utilisateur) n'avait **aucun hook i18n** :
**9 chaînes FR figées** affichées en TOUTES langues — rupture Prisme. Deux visibles
(titre + 6 onglets) et deux a11y (sr-only + aria-label du select mobile).

| Chaîne | Avant (FR figé) | Clé |
|--------|-----------------|-----|
| Titre dialog | `Paramètres et Configuration` | `configModal.title` |
| Onglet user | `Profil utilisateur` | `configModal.tabs.user` |
| Onglet langue | `Langues & Traduction` | `configModal.tabs.language` |
| Onglet thème | `Apparence` | `configModal.tabs.theme` |
| Onglet stats | `Statistiques` | `configModal.tabs.stats` |

## Correctifs
- `useI18n('settings')` câblé ; 9 chaînes → `t('configModal.*', '<EN fallback>')`.
- Nouveau bloc `settings.configModal` (9 clés) ×4 locales — **additif** sous la
  racine `settings` existante, **pas de nouveau namespace** (le hook
  `useI18n('settings')` dé-emballe la racine `settings` puis navigue `configModal.*`).
- **Fallbacks EN en 2e arg** (anti-flash, leçon 50w) : titre + 6 onglets sont
  **visibles** → fallback anglais évite tout flash de clé brute pendant le load async.

## Décisions
- **Bloc `configModal` dédié** plutôt que réutilisation de `settings.tabs.*` existant :
  ce dernier emploie un libellé court (« Profile », « Translation »…) ≠ des libellés
  riches actuels (« Profil utilisateur », « Langues & Traduction »). Réutiliser aurait
  **changé le texte affiché** — l'objectif ici est une i18n **fidèle** (même UX, juste
  localisée), pas une refonte de libellés. Bloc dédié = zéro changement visuel FR.
- Test existant `__tests__/components/settings/config-modal.test.tsx` mis à jour :
  mock déterministe de `@/hooks/use-i18n` (`t(key, fallback) => fallback`) + assertions
  basculées sur les libellés **anglais** (comportement réel hors traductions chargées).
  **22/22 verts**.
