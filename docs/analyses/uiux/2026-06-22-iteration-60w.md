# Itération 60w — Analyse UI/UX (Web)

**Date** : 2026-06-22
**Périmètre** : Application web (`apps/web`) **exclusivement**
**Surface** : `components/settings/config-modal.tsx` (modale globale Paramètres & Configuration)

## Contexte

La routine UI/UX web avance par petits lots bornés et orthogonaux (forte contention
des agents parallèles sur feed/reels/modales 57w–59w). Le différé borné documenté en
fin de 59w désignait explicitement `config-modal.tsx` comme **bon candidat 60w** :
libellés FR figés en TOUTES langues sur une surface user-facing.

## Constat (rupture Prisme Linguistique)

`ConfigModal` est la modale de configuration globale (montée par
`components/conversations/create-link-button.tsx`). Elle **n'avait AUCUN hook i18n** :
9 chaînes user-facing étaient codées en dur en français et s'affichaient telles quelles
quelle que soit la langue d'interface choisie par l'utilisateur — violation directe du
Prisme (« le contenu de l'interface doit être consommé dans la langue configurée »).

| # | Chaîne FR figée | Type | Visible / a11y |
|---|-----------------|------|----------------|
| 1 | `Paramètres et Configuration` | Titre `DialogTitle` | Visible |
| 2 | `Profil utilisateur` | Libellé onglet `user` | Visible (sidebar + select mobile) |
| 3 | `Langues & Traduction` | Libellé onglet `language` | Visible |
| 4 | `Apparence` | Libellé onglet `theme` | Visible |
| 5 | `Statistiques` | Libellé onglet `stats` | Visible |
| 6 | `Notifications` | Libellé onglet `notifications` | Visible |
| 7 | `Confidentialité` | Libellé onglet `privacy` | Visible |
| 8 | `Sélectionner une section` | `<label sr-only>` du select mobile | a11y (lecteur d'écran) |
| 9 | `Section des paramètres` | `aria-label` du `<select>` mobile | a11y (lecteur d'écran) |

Chaque libellé d'onglet apparaît **deux fois** (sidebar desktop + `<option>` du select
mobile), aggravant l'incohérence visuelle pour un utilisateur non-francophone.

## Correction appliquée

- Bloc i18n dédié `settings.configModal` (×4 locales en/fr/es/pt, parité vérifiée) :
  - `title`, `selectSection`, `sectionAriaLabel`
  - `tabs.{user,language,theme,stats,notifications,privacy}`
- `useI18n('settings')` câblé dans le composant ; chaque libellé passe par
  `t('configModal...', '<fallback EN>')` — **fallback anglais en 2e argument**
  (anti-flash de clé brute pendant le chargement async, leçon 50w).
- Pourquoi un bloc **dédié** plutôt que réutiliser `settings.tabs.*` existant : les
  libellés diffèrent sémantiquement (`Profil utilisateur` ≠ `tabs.profile`="Profil" ;
  `Langues & Traduction` ≠ `tabs.translation`="Traduction" ; `Apparence` ≠
  `tabs.theme`="Thème"). Réutiliser `tabs.*` aurait changé le texte affiché et couplé
  deux surfaces distinctes. Le bloc dédié préserve le wording d'origine à l'identique.

### Test

`__tests__/components/settings/config-modal.test.tsx` existait déjà et assertait sur
les libellés FR **en synchrone**. Le hook `useI18n` réel résout les libellés de façon
**asynchrone** (rendu initial = fallback EN), ce qui aurait cassé les assertions. Le
hook est désormais **mocké** dans ce test pour résoudre les clés `configModal` vers les
libellés FR attendus → les assertions de comportement (navigation onglets, surbrillance,
a11y, props enfants) restent valides et déterministes.

## Vérifications

- JSON des 4 locales valide (`JSON.parse`) + parité de clés `configModal` (en/fr/es/pt).
- Grep résiduel FR figé sur le composant = 0.
- `ConfigModal` n'a qu'un seul appelant (`create-link-button.tsx`) — aucune régression
  d'API (props inchangées).
