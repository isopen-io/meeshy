# UI/UX Analysis — Iteration 86i (2026-07-01)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android couverts par d'autres agents).
Thème : **accessibilité — Dynamic Type + en-têtes VoiceOver sur `AboutView`**. Migration
mécanique des polices figées `.font(.system(size:))` vers `MeeshyFont.relative(...)` (scaling
Dynamic Type) sur l'écran « À propos », + trait `.isHeader` sur les en-têtes de section pour
la navigation au rotor VoiceOver.

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de
build de validation (SwiftUI ne compile pas sous Linux). **Aucun test neuf** : sweep
typographique pur + traits a11y, aucune logique modifiée (parité 55i / 71i / 72i / 74i).
`MeeshyFont` est déjà en scope (`import MeeshyUI` présent ; `MeeshyFont.relative` est un
`public static func` de `MeeshyUI/Theme/Accessibility.swift`, déjà consommé par
`ReportUserView`, `TwoFactorSetupView`, `SettingsView`, `ConversationView`…).

## Contexte / point de départ
`AboutView.swift` (283 lignes) est l'écran « À propos » : logo animé, version, sections
Informations / Description / Fonctionnalités / Liens / Copyright. Tous les libellés étaient
déjà correctement i18n'd (`String(localized:defaultValue:bundle:)`) mais **16 sites** de
police fixaient une taille absolue `.font(.system(size:))` — l'écran ignorait entièrement le
réglage Dynamic Type de l'utilisateur (rupture directe de la règle a11y CLAUDE.md « Use
semantic fonts, NEVER use fixed font sizes for body text »).

### VoiceOver — en-têtes de section navigables au rotor
`sectionHeader(title:icon:color:)` combine désormais ses enfants en un seul élément, expose
le `title` **non-uppercased** (le rotor lit le libellé naturellement, sans épeler les majuscules)
et porte `.accessibilityAddTraits(.isHeader)` → l'utilisateur VoiceOver navigue de section en
section via le rotor « En-têtes », cohérent avec le titre nav qui portait déjà `.isHeader`.

## Périmètre délibérément exclu (ne pas re-flagger)

### `fieldIcon` (ligne 214) — glyphe 14pt gardé figé
`fieldIcon(_:color:)` rend un glyphe SF Symbol **dans un badge de dimension fixe 28×28**
(`.frame(width: 28, height: 28)` + fond `RoundedRectangle`). Laisser la police scaler
déborderait/tronquerait le glyphe hors du badge fixe. Précédent 74i (`StatRing` anneau fixe
60×60, `ArcGauge`) : les glyphes contraints dans un conteneur de dimension fixe restent figés.
**Décision : conservé `.font(.system(size: 14, weight: .medium))`.** (Un passage à `@ScaledMetric`
sur le badge + le glyphe serait la seule alternative propre — hors-scope de ce sweep léger.)

### Couleurs laissées intactes (hors-scope palette)
- **Ladder décoratif des teintes de section** : `accentColor = "45B7D1"` (Informations),
  `MeeshyColors.indigo600Hex` (Description), `"F8B500"` (Fonctionnalités), `MeeshyColors.infoHex`
  (Liens). Ce sont des accents décoratifs par-section (analogues à l'accent déterministe de
  conversation) — décision « ladder catégoriel = charte unique différée » (69i/70i/78i).
- **Logo** : `isDark ? .white : Color(hex: "1C1917")` — traitement de marque du logo animé.
- **Checkmark `#4ADE80`** (ligne 255) : proche mais **non exact** du token `success = #34D399`.
  Le différé 78i le classe explicitement « à auditer un par un **avec vérif visuelle** » — pas
  d'infra snapshot pour cet écran → swap couleur différé, hors de ce lot Dynamic Type.
