## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **accessibilité Dynamic Type — migration d'une surface primaire jamais traitée**
(`GlobalSearchView`). Reprise de la passe « éradication des `.font(.system(size:))` figés »
amorcée 32/42/44b/45i/46i (`FriendRequestListView`), via l'atome SDK déjà éprouvé
`MeeshyFont.relative(_:weight:design:)` (`MeeshyUI/Theme/Accessibility.swift`). Itération
bornée et « épurée » : **1 fichier**, swaps mécaniques 1:1, aucune surcharge ajoutée.

La CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur) sert de vérification de
build (pas de build local SwiftUI sur Linux ; le SDK non plus). XcodeGen globe les `.swift`
sous `Meeshy/` → aucune édition `project.pbxproj` requise.

## Contexte / point de départ
- L'atome `MeeshyFont.relative(size, weight:design:)` mappe une taille point legacy vers le
  `Font.TextStyle` relatif le plus proche (`textStyle(for:)`) → la police **scale avec le
  réglage Dynamic Type de l'utilisateur** tout en préservant weight/design. Migration =
  swap mécanique : `.font(.system(size: 15, weight: .medium))` →
  `.font(MeeshyFont.relative(15, weight: .medium))`.
- `GlobalSearchView` (recherche globale, surface primaire à fort trafic) n'avait **jamais**
  reçu de passe Dynamic Type : 33 occurrences `.font(.system(size:))` toutes figées
  (listée uniquement en « différé » par 44/45i/46i/48i, jamais traitée — `MeeshyFont.relative`
  count = 0 avant cette itération). Apple HIG : « NEVER use fixed font sizes for body text »
  (cf. `apps/ios/CLAUDE.md` § Accessibility).

## iOS Findings

### Dynamic Type figé — `GlobalSearchView` (33 occurrences `.font(.system(size:))`)
**31 migrées** vers `MeeshyFont.relative(...)` (mapping iter-32/42 via `textStyle(for:)` :
18→`.body`, 15→`.subheadline`, 14→`.subheadline`, 13→`.footnote`, 12→`.caption`,
11→`.caption2`, weight/design préservés) :

- **Header** : chevron retour (18), icône loupe (15), champ de saisie (15).
- **Barre d'onglets** : icône onglet (12), libellé onglet (13, weight conditionnel `isSelected`).
- **États** : indicateur « Recherche en cours » (14), en-tête + lignes « Recherches
  récentes » (14/14/12/14/11), états vides via `EmptyStateView` (déjà relatifs).
- **Lignes de résultats messages** : nom conv (14), time-ago (11), nom expéditeur (12),
  contenu (13) + **run surligné de l'`AttributedString`** (`highlightedText`, 13 bold) — migré
  aussi pour que le mot surligné scale en cohérence avec le texte alentour.
- **Lignes conversations** : `ConversationTitleLabel(font:)` (14 bold), icône+libellé de type
  (11/12), compteur de membres (12), libellés `lastMessage` (icône 11 + texte 13 *italic*
  pour caché/vue-unique/expiré, preview 13).
- **Lignes utilisateurs** : nom (14), `@username` (13), badge « En ligne » (11).

### Écarté (et pourquoi — ne pas re-flagger comme « oubli »)
**2 badges numériques laissés figés** (exception documentée inline, précédent : héros 48pt
de `FriendRequestListView` en 46i) :
- **Badge de comptage d'onglet** (`size: 9, .bold`) : micro-pastille positionnée en
  `.overlay(.topTrailing)` via `.offset(x:10,y:-8)` + `.fixedSize()`. La faire scaler la
  ferait déborder de son ancre d'overlay.
- **Badge non-lus** (`size: 11, .bold`) dans la ligne conversation : compteur compact
  (convention iOS) — gardé hors Dynamic Type pour que la capsule reste « pill-tight ».

## Conforme (vérifié — ne pas re-flagger)
- A11y déjà conforme et non touchée : tous les boutons/lignes portent déjà
  `.accessibilityLabel`/`.accessibilityHint`/`.accessibilityAddTraits`, icônes décoratives
  `.accessibilityHidden(true)`. La migration police ne change aucun label.
- Couleurs/gradients inchangés. Le **fond du champ de recherche du header** a depuis été
  migré en `.adaptiveGlass(...)` par l'itération **68i** (Glass) — convergence sans conflit :
  68i touche le `.background`/chrome, 54i touche les `.font` ; les deux coexistent dans `main`.
- `EmptyStateView` (états vides) utilise déjà des styles relatifs en interne.
