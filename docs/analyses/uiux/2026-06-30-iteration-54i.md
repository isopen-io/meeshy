## iOS Findings

### i18n/a11y — 3 labels VoiceOver figés en français (`InviteFriendsSheet`)
Le fichier utilise `String(localized:…, bundle: .main)` partout (titre, boutons, `common.close`),
mais **3 `accessibilityLabel` étaient des littéraux français** → VoiceOver annonçait du français
à TOUS les utilisateurs (EN/ES/DE/PT) = **rupture Prisme + bug a11y** :
- L.223 « Copier le lien d'invitation » (bouton copier)
- L.310 « Partager le lien d'invitation » (bouton partager)
- L.341 « Personnaliser les options du lien » (bouton options)

**Correctif** : `String(localized: "invite.a11y.{copyLink,shareLink,customizeOptions}",
defaultValue: "<EN>", bundle: .main)` — pattern a11y du fichier (cf. L.111 `common.close`).
Les 3 clés **ajoutées au String Catalog** `apps/ios/Meeshy/Localizable.xcstrings`
(`sourceLanguage: fr`) avec traductions **×5 langues** (de, en, es, fr, pt-BR) — la valeur `fr`
reprend **exactement** le littéral d'origine (zéro régression francophone), + 4 langues neuves.
Édition catalog en préservant l'ordre/format Xcode (`" : "`, indent 2, pas de tri) → diff minimal.

Thème : **épuration palette — consolidation des teintes sémantiques « flat-UI » hors-charte
vers les tokens `MeeshyColors`** (Single Source of Truth, charte Indigo Meeshy). Itération
bornée et cohérente (« logique épurée ») : 3 fichiers, 4 occurrences, **swap de tokens pur,
zéro changement de comportement**. Continuité directe du différé « palette tokens » de 53i.

Vérification : CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur) = seule vérif de
build (pas de SwiftUI sur Linux). `MeeshyColors` est accessible sans import additionnel via
`@_exported import MeeshyUI` (les 3 fichiers importent déjà `MeeshySDK`).

## Contexte / point de départ
La charte Meeshy (`apps/ios/CLAUDE.md` § Semantic State Colors) définit des tokens sémantiques
adaptés à l'identité de marque :
- `MeeshyColors.success` = `#34D399` (sent/online/confirmed), `successDeep` = `#10B981`
- `MeeshyColors.info` = `#60A5FA` (information/aide)

Plusieurs vues conservaient encore des **verts/bleus « flat-UI » hors-charte** (`#2ECC71`,
`#3498DB`, `#27AE60` — palette Nephew/flat-ui-colors, **pas** la marque) là où la sémantique
est **non ambiguë** (téléphone = vert positif, e-mail = bleu info, partage = vert positif,
épingle de localisation = vert). Ces littéraux dupliquent une valeur de marque sans passer par
la source de vérité → dérive de palette + incohérence dark/light potentielle.

## iOS Findings

### Épuration — `ContactCardView` (carte contact dans une bulle)
- Icône téléphone `phone.fill` : `Color(hex: "2ECC71")` → `MeeshyColors.success`.
- Icône e-mail `envelope.fill` : `Color(hex: "3498DB")` → `MeeshyColors.info`.
Sémantique exacte (vert = ligne téléphonique active/positive, bleu = info e-mail) ; aligne la
carte sur la charte. Résout le différé « palette tokens » explicitement listé par 53i pour
cette carte (53i n'avait migré que le **fond** en Liquid Glass, pas ces 2 teintes).

### Épuration — `AffiliateView` (ligne de lien d'affiliation)
Bouton **partager** `square.and.arrow.up` : `Color(hex: "2ECC71")` → `MeeshyColors.success`.
**Cohérence de paire** : le bouton **supprimer** adjacent utilise déjà `MeeshyColors.error` ;
les deux actions de la même rangée passent désormais par des tokens sémantiques (vert/rouge).

### Épuration — `FeedView+Attachments` (vignette de pièce jointe de type localisation)
Dégradé de la tuile localisation : `[Color(hex: "2ECC71"), Color(hex: "27AE60")]` →
`[MeeshyColors.success, MeeshyColors.successDeep]` (clair → profond, ordre préservé).
**Cohérence intra-écran** : dans le **même composer**, le bouton « partager la position »
(`location.fill`) utilise déjà `MeeshyColors.success` → la vignette de prévisualisation parle
maintenant le même vert de marque que son bouton déclencheur.

## Périmètre délibérément exclu (ladder catégoriel arc-en-ciel — différé charte)
**NON touché** : le **ladder catégoriel** des boutons de pièces jointes du composer
(`FeedView.swift` / `FeedView+Attachments.swift`) — emoji `#F8B500` (or), fichier `#9B59B6`
(violet). Ce sont des **couleurs par catégorie** (chaque type d'attachement a sa teinte) :
un swap mono-token naïf détruirait la distinction visuelle. Conformément au différé récurrent
« ladder catégoriel arc-en-ciel », ils relèvent d'une **décision de charte unique** (avec
`MediaTypes.speakerPalette`, `NotificationListView.color`, `TagInputView`), pas de cette
épuration sémantique ponctuelle. **Ne pas re-flagger isolément.**

## Conforme (vérifié — ne pas re-flagger)
- `MeeshyColors.success`/`.info`/`.successDeep` sont des littéraux fixes (non theme-aware) :
  le swap **n'introduit pas** d'adaptivité dark/light nouvelle — le gain est la **cohérence
  de palette / source unique** (principe « Single Source of Truth »), pas un fix dark-mode.
- A11y des 3 composants inchangée (labels/hints déjà présents) ; gestes, layout, tailles de
  police, formes Liquid Glass (53i) intacts. Swap de teinte d'icône uniquement.
- `attachment.thumbnailColor` (dégradé générique des autres types de pièces jointes) reste
  piloté par la donnée — hors périmètre.
