# UI/UX Analysis — Iteration 53i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **adoption native iOS 26 Liquid Glass — lot 3** sur la **couche d'overlay
« effets » d'appel** (`CallEffectsOverlay`), via l'atome SDK `adaptiveGlass`
(`packages/MeeshySDK/.../Compatibility/AdaptiveGlass.swift`), déjà éprouvé en 51i
(`FloatingCallPillView`, `ContextActionMenu`) et 52i (`MiniAudioPlayerBar`,
`MentionSuggestionPanel`). Itération bornée et cohérente (« logique épurée ») :
3 swaps 1:1 fidèles sur **3 surfaces flottantes sœurs**, aucune surcharge ajoutée.

La CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur iOS 18.2) sert de
vérification de build (pas de build SwiftUI/SDK sur Linux). XcodeGen globe les `.swift`
sous `Meeshy/` → aucune édition `project.pbxproj` requise.

## Contexte / point de départ
- 52i a soldé 2 des items « Glass adoption (reste) » différés par 51i
  (`MiniAudioPlayerBar` capsule + `MentionSuggestionPanel` bande). Base = `main` post #1076.
- La couche d'overlay des **effets pendant un appel** (`CallEffectsOverlay`) compose
  3 surfaces flottantes content-agnostic, toutes restées sur le « vieux »
  `.background(.ultraThinMaterial).clipShape(...)` — exactement le pattern que 52i a
  migré ailleurs. Ces 3 surfaces sont **sœurs** (siblings dans un `VStack`, ou panneau
  vs toolbar distincts au-dessus d'un backdrop `Color.black.opacity(0.25)`) → **aucun
  risque verre-dans-verre entre elles**.

## iOS Findings — `CallEffectsOverlay` (couche effets d'appel)

### Adoption 1 — `AudioEffectsPanel` → Liquid Glass natif (panneau, neutre)
Panneau flottant des effets audio en appel (`CallManager.shared`). Composait
`.background(.ultraThinMaterial).clipShape(RoundedRectangle(cornerRadius: .lg))`.
**Correctif** : `.background(.ultraThinMaterial)` → `.adaptiveGlass(in: RoundedRectangle(cornerRadius: .lg))`,
le `.clipShape(...)` existant conservé après (pattern `FloatingCallPillView`).
- **iOS 26+** : panneau en Liquid Glass natif **neutre** (pas de teinte marque — chrome
  OS, registre QuickType bar ; un panneau d'effets teinté accent lirait comme du contenu).
- **iOS < 26** : `.ultraThinMaterial` + hairline (fallback de l'atome).

### Adoption 2 — `VideoFiltersPanel` → Liquid Glass natif (panneau, neutre)
Idem pour le panneau des filtres vidéo. **Même swap 1:1.**
- **Subtilité glass-in-glass évitée** : `VideoFiltersPanel` **imbrique** une sous-section
  `VideoFilterControlView` qui possède son **propre** `.background(.ultraThinMaterial)`.
  On migre **uniquement** le panneau parent au verre ; la sous-section reste en
  `.ultraThinMaterial` → **matériau SUR verre** (pattern HIG recommandé : vibrancy
  fills sur le verre), **jamais verre-dans-verre** (qui clipperait). `VideoFilterControlView`
  n'est utilisé qu'**ici** (vérifié : aucun autre site) → on ne le migre pas.

### Adoption 3 — `CallEffectsOverlay.secondaryToolbar` → Liquid Glass natif (capsule, neutre)
La toolbar capsule flottante (boutons Effets/Filtres) composait
`.background(.ultraThinMaterial).clipShape(Capsule())`. **Correctif** :
`.adaptiveGlass(in: Capsule())` avant le `.clipShape(Capsule())` existant — **1:1 avec
`FloatingCallPillView` et `MiniAudioPlayerBar` (52i)**. Les cercles d'état actif des
boutons (indigo) restent des remplissages SUR le verre (HIG).

## Écarté (et pourquoi — ne pas re-flagger comme « oubli »)
- **`VideoFilterControlView`** (sous-section imbriquée dans `VideoFiltersPanel`) : laissé
  en `.ultraThinMaterial` **délibérément** — matériau-sur-verre = correct HIG ; le migrer
  créerait du verre-dans-verre. **Ne pas migrer tant qu'il reste imbriqué.**
- **`CallView` (l.681, l.951)** : surfaces de la vue d'appel principale (gros fichier) —
  lot dédié si jugé utile, hors périmètre « épuré » de 53i.
- **`MessageInfoSheet` / `StatusBubbleOverlay` / `UniversalComposerBar` /
  `MessageOverlayMenu` (l.918)** : tous stylés avec un **stroke en dégradé accent** (même
  cas que `ContactCardView`, écarté en 52i). Le modèle mono-teinte d'`adaptiveGlass` ne
  reproduit pas le stroke dégradé (perte de nuance). Préserver le style existant > forcer
  l'atome. **Laissés tels quels.**

## Conforme (vérifié — ne pas re-flagger)
- Pattern d'appel `adaptiveGlass(in: shape).clipShape(shape)` aligné sur l'existant
  (`FloatingCallPillView`, `MiniAudioPlayerBar`).
- Les 3 fichiers importent déjà `MeeshyUI` (où vit `adaptiveGlass`).
- A11y de la toolbar déjà conforme (`accessibilityValue` on/off + `accessibilityHint`
  open/close localisés sur chaque bouton). Non touchée.
- `AudioEffectsPanelTests` / comportement panneaux (sélection effet, debounce, routing
  `CallManager`) → inchangés (swap purement visuel).

## Anti-repetition check
Vérifié contre 52i (`MiniAudioPlayerBar` + `MentionSuggestionPanel` — jamais
`CallEffectsOverlay`/`AudioEffectsPanel`/`VideoFiltersPanel`), 51i (`ContextActionMenu` +
fallback teinté), 49i/48i (palettes chrome). Aucune itération antérieure n'avait migré la
couche effets d'appel. Solde 3 surfaces de plus dans « Glass adoption (reste) ».
