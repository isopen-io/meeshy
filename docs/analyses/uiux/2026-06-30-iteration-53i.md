# UI/UX Analysis — Iteration 53i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **épuration palette — élimination des hex hors-marque dans les composants de
contexte message**, alignement strict sur les tokens sémantiques `MeeshyColors`. Itération
bornée et cohérente (« logique épurée ») : 2 fichiers, 4 swaps 1:1 hex → token, zéro
changement de layout.

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur iOS 18.2)
sert de build de validation (pas de build SwiftUI local sur Linux). XcodeGen globe les
`.swift` sous `Meeshy/` → aucune édition `project.pbxproj` requise.

## Contexte / point de départ
La charte Meeshy (`apps/ios/CLAUDE.md` § Key Patterns / SDK CLAUDE.md § Visual Identity)
pose deux règles dures :
- **« ALL conversation-context components MUST use `accentColor`, never hardcode colors »**
- **« Semantic colors (error, success, …) remain static via `MeeshyColors` »** ; « New code
  MUST use the Indigo scale or semantic names. »

Le token canonique du statut « lu » existe déjà et n'était pas utilisé :
`MeeshyColors.readReceipt = indigo400 (#818CF8)` — « Read indicator » dans le tableau
Semantic State Colors.

Deux composants de **contexte message** composaient encore des hex hors-palette codés en
dur (vestiges « system gray » iOS / bleu WhatsApp / verts-bleus génériques) :

| Fichier | Ligne | Hex codé en dur | Rôle |
|---|---|---|---|
| `MessageInfoSheet` | 257 | `#8E8E93` | icône statut **Distribué** (gris système iOS) |
| `MessageInfoSheet` | 270 | `#34B7F1` | icône statut **Lu** (bleu type WhatsApp) |
| `ContactCardView` | 67 | `#2ECC71` | icône **téléphone** (vert générique) |

## iOS Findings

### Épuration — `MessageInfoSheet` timeline de livraison → tokens sémantiques
La timeline « Statut de livraison » (Envoyé → Distribué → Lu) mélangeait un token
(`theme.textMuted` pour « Envoyé ») et **deux hex hors-marque** pour « Distribué » et « Lu ».
- **Distribué** : `#8E8E93` (gris système iOS, étranger à l'échelle neutre Meeshy) →
  `MeeshyColors.neutral400` (`#9CA3AF`), le neutre de marque le plus proche perceptuellement.
- **Lu** : `#34B7F1` (bleu « ticks bleus » WhatsApp) → `MeeshyColors.readReceipt`
  (`#818CF8`, indigo400) — c'est **exactement** le token désigné pour l'indicateur de lecture
  dans la charte. Mapping sémantique parfait : le statut « Lu » lit désormais en bleu de
  marque cohérent avec la bande de drapeaux et les accusés de réception du reste de l'app.

### Épuration — `ContactCardView` (carte contact dans une bulle) → tokens sémantiques
La carte contact partagée (rendue **dans** une bulle de message) teintait ses lignes
téléphone/email avec des hex génériques :
- **Téléphone** : `#2ECC71` → `MeeshyColors.success` (`#34D399`). Le vert de marque pour
  l'affordance d'appel.
- **Email** : `#3498DB` → `MeeshyColors.info` (`#60A5FA`). Le bleu d'information de marque
  pour l'affordance courriel.

Le **stroke en dégradé accent** (`accentColor` 0.3→0.1) et le fond `.ultraThinMaterial`
de la carte sont **préservés tels quels** — seule la teinte des deux icônes de ligne change.

## Pourquoi ces mappings (et pas un autre)
- **dark/light safe** : les tokens `MeeshyColors` sémantiques sont conçus theme-cohérents
  (ils ne sont jamais re-teintés par mode). Les hex hors-marque, eux, étaient des constantes
  aveugles au thème — l'épuration améliore la cohérence dark/light sans branche `isDark`.
- **`readReceipt` plutôt que `info`** pour « Lu » : la charte réserve `readReceipt` (indigo400)
  à l'indicateur de lecture ; l'utiliser ici unifie le langage visuel « lu » sur toute l'app.
- **`neutral400` plutôt que `textMuted`** pour « Distribué » : conserve la **hiérarchie**
  visuelle existante (Envoyé `textMuted` < Distribué neutre plus présent < Lu accent), au lieu
  d'aplatir Distribué sur la même teinte qu'Envoyé.

## Périmètre délibérément exclu (ne pas re-flagger comme « oubli »)
Les autres occurrences `#2ECC71`/`#3498DB` du repo sont des **accents-feature délibérés**,
**hors** contexte message — laissés intacts pour garder le lot épuré et orthogonal :
- `AffiliateView` / `AffiliateCreateView` (accent feature affiliation), `RootView` menu
  Communautés, pin de **localisation** vert (`FeedView+Attachments`, `ComposerModels`,
  `UniversalComposerBar+Attachments` — convention cartographique), `UserStatsView` /
  `StatsTimelineChart` (accent stats), `ReportUserView` / `SupportView` / `PrivacySettingsView`
  (accents de section passés en `tint:` à `theme.surfaceGradient`/`border`, déjà theme-aware).
Ces surfaces relèvent d'un éventuel **ladder catégoriel arc-en-ciel** (décision charte unique,
différée), pas d'une épuration au cas-par-cas.

## Conforme (vérifié — ne pas re-flagger)
- A11y des deux composants déjà conforme : `ContactCardView` a `accessibilityElement(.combine)`
  + label/hint localisés ; les lignes de la timeline `MessageInfoSheet` portent leur label.
  Non touchée.
- Le stroke dégradé accent + le `.ultraThinMaterial` de `ContactCardView` sont préservés
  (52i avait justement écarté cette carte du swap glass pour ne pas perdre le dégradé — ce
  lot ne touche **que** la teinte des icônes, cohérent avec cette décision).

## Anti-repetition check
Vérifié contre 52i (glass `MentionSuggestionPanel`/`MiniAudioPlayerBar` — a explicitement
**laissé `ContactCardView` tel quel**, sans toucher ses hex d'icônes), 68i (glass champ
recherche + mention, a11y skeleton), 51i (`ContextActionMenu`), 49i/48i (palettes chrome SDK
/ apps). **Aucune** itération antérieure n'avait migré les hex de statut de `MessageInfoSheet`
ni les icônes téléphone/email de `ContactCardView` vers les tokens sémantiques. Lot inédit.
