# UI/UX Analysis — Iteration 53i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **adoption native iOS 26 Liquid Glass sur la capsule flottante du quick-reaction
picker** (`EmojiReactionPicker`, composant partagé SDK `MeeshyUI`) — **lot 3** de la série
Glass entamée en 51i (`ContextActionMenu`) et 52i (`MentionSuggestionPanel` +
`MiniAudioPlayerBar`). Itération bornée et « épurée » : un seul composant, l'interaction
cœur réaction, zéro changement de comportement.

La CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de vérification
de build — le SDK/UI ne build pas localement sur Linux (SwiftUI absent).

## Contexte / point de départ
- `adaptiveGlass` (`MeeshyUI/Compatibility/AdaptiveGlass.swift`) est l'atome qui gate le
  vrai `glassEffect` iOS 26 et dégrade en `.ultraThinMaterial` flouté avant. Adopté à ce
  jour par : `FloatingCallPillView`, `CallView`, `ConversationListView+Overlays`,
  `ReelsPlayerView`, `IncomingCallView`, `ContextActionMenu` (51i), `MentionSuggestionPanel`
  + `MiniAudioPlayerBar` (52i), `StoryComposerView`, `UserProfileSheet+Header`.
- `EmojiReactionPicker.stripBackground` (la **capsule flottante** derrière la bande
  d'emojis) composait encore manuellement le « vieux » style :
  - dark : `Capsule().fill(.ultraThinMaterial)` + voile `Color.black.opacity(0.2)` +
    `stroke(.white 0.15)` + `shadow(.black 0.3, r12, y4)` ;
  - light : `Capsule().fill(.regularMaterial)` + `stroke(.gray 0.15)` +
    `shadow(.black 0.08, r8, y2)`.
  C'est exactement le pattern que Liquid Glass remplace. Ce composant est **partagé** et
  rendu sur **4 surfaces** : le strip inline long-press (`ConversationView+MessageRow`),
  l'overlay long-press (`MessageOverlayMenu`), la sidebar story (`StoryViewerView+Sidebar`)
  et les réactions par pièce jointe (`BubbleStandardLayout+Media`). Une seule migration
  bénéficie aux quatre.

## iOS Findings

### Adoption — `EmojiReactionPicker` capsule → Liquid Glass natif (iOS 26)
Remplacement de `.background(stripBackground)` (sur `quickEmojiStrip` **et**
`scrollableQuickEmojiStrip`) par un `ViewModifier` partagé `QuickReactionStripChrome` :
- **iOS 26+** : `.adaptiveGlass(in: Capsule())` **neutre** (pas de teinte marque — chrome
  OS, comme le QuickType bar). Le verre natif **échantillonne le contenu derrière** la
  capsule → elle s'adapte d'elle-même à n'importe quel fond (liste de messages, scrim du
  long-press, story plein écran) **sans le voile sombre manuel**. L'**ombre d'élévation**
  est conservée (modifier séparé, pilotée par `style` : dark `0.3/r12/y4`, light
  `0.08/r8/y2`) — cohérent avec l'exception « no-shadow » documentée pour le chrome
  flottant (51i `ContextActionMenu`, `FloatingCallPillView`).
- **iOS < 26** : **rendu identique au code actuel** (matériau + voile + liseré + ombre
  pilotés par `style`), byte-for-byte → **zéro régression** sur OS antérieur.

### Pourquoi un gate `if #available` local plutôt que `adaptiveGlass` nu
`EmojiReactionPicker` expose un paramètre **`style: .dark/.light`** qui **force** une tonalité
indépendamment du `colorScheme` système (la sidebar story passe `style: .dark` alors que le
système peut être en clair, pour rester sombre par-dessus la story sombre). L'atome
`adaptiveGlass` neutre est piloté par le `colorScheme` système, pas par `style` : l'utiliser
nu aurait, **en pré-iOS-26 + mode clair système + contexte forcé `.dark`** (story), rendu un
matériau clair par-dessus un fond sombre — une régression. Le gate local préserve donc le
matériau `style`-driven sur le fallback ; sur iOS 26 le problème disparaît (le verre
échantillonne le contenu, `style` devient sans objet pour le fond).

## Conforme (vérifié — ne pas re-flagger)
- **Contrôles internes** (`expandButton` cercle `white.opacity(0.15)`, tuiles emoji) : restent
  des **fills de vibrancy**, pas du verre → pas de glass-in-glass (HIG), identique au choix
  52i `MiniAudioPlayerBar`.
- **Cascade d'entrée** `WaveTileModifier` (vague sinusoïdale gauche→droite), haptique
  `reactToEmoji`, `scale`/`scrollable`/fade-mask : inchangés.
- **Paramètre `style`** : toujours nécessaire (pilote `expandButton`, l'ombre, le fallback) —
  pas de paramètre mort.

## Différés (volontaires — NE PAS re-flagger comme manquants)
- `EmojiFullPickerSheet.sheetBackground` et `EmojiKeyboardPanel` (mêmes fichier/série) :
  **surfaces de contenu** (feuille bas d'écran scrollable / panneau clavier), pas des chromes
  flottants — le verre derrière du contenu scrollable nuit à la lisibilité (HIG). Laissés en
  matériau, comme `MessageOverlayMenu.panelBackground` (feuille détail) et
  `ContactCardView`/`LocationPickerView` (laissés en 52i). Décision design, **pas** une dette.
- `BubbleStandardLayout+Media:374` enveloppe le picker dans son propre
  `.background(.ultraThinMaterial, in: RoundedRectangle(16))` au call-site — wrapper de
  réactions par pièce jointe, hors scope capsule ; à arbitrer plus tard si on veut l'unifier.

## Anti-repetition check
Vérifié contre : 51i (`ContextActionMenu` — capsule différente, app-side ; `AdaptiveGlass`
fallback teinté), 52i (`MentionSuggestionPanel` + `MiniAudioPlayerBar` — laissait
explicitement `ContactCardView`/`LocationPickerView`, ne touchait pas `EmojiReactionPicker`),
49i/48i (palette, pas le glass). Aucune itération antérieure n'avait migré la capsule du
reaction picker. La série Glass progresse par lots flottants/content-agnostic bornés.

## Différés (pour itérations suivantes — iOS)
- Reste Glass content-agnostic : `ContactCardView`, `LocationPickerView` (réévaluer : sheet

- Polices figées : `FriendRequestListView` (11), grandes surfaces (`ConversationInfoSheet`,
