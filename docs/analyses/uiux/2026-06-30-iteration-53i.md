# UI/UX Analysis — Iteration 53i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **adoption native iOS 26 Liquid Glass — lot 3** sur les surfaces **flottantes**
restées sur le « vieux » `.ultraThinMaterial`, via l'atome SDK `adaptiveGlass` déjà
éprouvé (51i `ContextActionMenu`, 52i `MentionSuggestionPanel` + `MiniAudioPlayerBar`).
Itération bornée et cohérente (« logique épurée ») : 2 fichiers, 4 surfaces, swaps fidèles
1:1 (fallback identique pré-iOS-26, ombres et liseré conservés).

La CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de vérification
de build (pas de build local SwiftUI sur Linux ; le SDK non plus). XcodeGen globe les
`.swift` sous `Meeshy/` → aucune édition `project.pbxproj` requise.

## Contexte / point de départ
- L'atome `adaptiveGlass(in:tint:interactive:)`
  (`packages/MeeshySDK/Sources/MeeshyUI/Compatibility/AdaptiveGlass.swift`) gate le vrai
  `glassEffect` iOS 26 et dégrade proprement pré-26 (`.ultraThinMaterial` + liseré, +
  teinte empilée quand fournie — correctif de fidélité 51i). API : modificateur de `View`,
  à appliquer EN DERNIER (après le sizing).
- Pattern d'appel canonique établi (`FloatingCallPillView`, `MiniAudioPlayerBar`) :
  `.adaptiveGlass(in: shape).clipShape(shape)[.overlay(stroke)].shadow(...)`.
- Les surfaces ciblées composaient encore `.background(RoundedRectangle(...).fill(.ultraThinMaterial).shadow(...))`.

## iOS Findings

### Adoption — `LocationPickerView` : 3 panneaux flottants sur la carte (neutre)
Le sélecteur de lieu superpose **3 surfaces flottantes au-dessus d'une `Map`** :
1. **Barre de recherche** (haut) — `RoundedRectangle(12).fill(.ultraThinMaterial).shadow`.
2. **Dropdown de résultats** (sous la barre) — `RoundedRectangle(12).fill(.ultraThinMaterial).shadow`.
3. **Carte d'action du bas** (adresse + « Ma position » + « Confirmer ») —
   `RoundedRectangle(20, continuous).fill(.ultraThinMaterial).shadow`.

**Correctif** : `.background(shape.fill(.ultraThinMaterial).shadow)` →
`.adaptiveGlass(in: shape).clipShape(shape).shadow(...)`.
- **iOS 26+** : verre Liquid Glass natif **neutre** (pas de teinte) — ces panneaux flottent
  au-dessus d'une carte (chrome de navigation, registre OS) ; une teinte accent les ferait
  lire comme du contenu. Choix cohérent avec la barre QuickType-like de 52i.
- **iOS < 26** : `.ultraThinMaterial` + liseré hairline (fallback de l'atome) — visuel
  pratiquement identique à l'existant.
- `clipShape` ajouté : sur le dropdown il garantit que les `Divider`/dernière ligne ne
  débordent pas des coins arrondis du verre. Les 2 boutons internes de la carte (CTA accent
  en dégradé + bouton « Ma position » teinté) restent des **remplissages SUR le verre**
  (Apple HIG : ne pas imbriquer du verre dans du verre).

### Adoption — `StatusBubbleOverlay` : bulle d'humeur flottante (liseré préservé)
La bulle « humeur/pensée » flotte au-dessus de l'écran, ancrée à un avatar (overlay, pas
imbriquée dans une bulle de message). Surface principale (`bubbleContent`) :
`RoundedRectangle(14, continuous).fill(.ultraThinMaterial).overlay(stroke dégradé avatar).shadow`.

**Correctif** : `.adaptiveGlass(in: RoundedRectangle(14, continuous)).clipShape(…).overlay(stroke).shadow(…)`.
- **iOS 26+** : verre natif, **avec préservation du liseré dégradé teinté avatar**
  (`status.avatarColor`→`white`, 0.5pt) en `overlay` + l'ombre d'élévation — **exactement
  l'idiome de `FloatingCallPillView`** (`adaptiveGlass` + `overlay(stroke gradient)` + `shadow`).
- **iOS < 26** : material + liseré + ombre, identique à l'existant.
- Contrairement à `ContactCardView` (écarté 52i), le liseré dégradé n'est **pas un blocage**
  ici : il est conservé tel quel en overlay au-dessus du verre, et la bulle est un overlay
  flottant (aucun risque glass-sur-glass d'une carte imbriquée dans une bulle).

## Écarté (et pourquoi — ne pas re-flagger comme « oubli »)
- **`StatusBubbleOverlay.thoughtCircle`** (les 3 petits cercles 4/7/10 pt du « fil de
  pensée » menant à la bulle) : laissés en `.ultraThinMaterial`. À cette taille la différence
  verre/material est imperceptible, et les rendre verre natif sans `GlassEffectContainer`
  introduirait un risque de clipping inutile entre atomes proches. Atomes décoratifs → hors
  périmètre.
- **`MessageInfoSheet.sectionBackground`** : utilisé par **5+ cartes de section** dans une
  sheet scrollable (sender, timeline, preview, consumption, receipts) au-dessus d'un fond
  solide `theme.backgroundPrimary`. Apple HIG : Liquid Glass est la couche **flottante**
  au-dessus du contenu, **pas** le fond de chaque carte d'une liste. Des cartes-material
  sur fond solide sont le traitement correct ici → **laissé tel quel** (ce n'est pas du
  chrome flottant).
- **`MessageOverlayMenu`** : reaction picker imbriqué (`EmojiReactionPicker` dans le
  panneau) → risque glass-sur-glass, nécessite un refactor préalable. Différé.
- **`InviteFriendsSheet`** : swap propre mais surface modale basse fréquence — à grouper
  avec un futur lot « sheets ». Différé.
- **`ContactCardView`** : déjà écarté 52i (carte dans bulle + stroke dégradé accent +
  risque glass-sur-glass). Inchangé.

## Conforme (vérifié — ne pas re-flagger)
- Pattern d'appel `adaptiveGlass` aligné sur l'existant (`FloatingCallPillView` :
  `.adaptiveGlass(in:).clipShape(…).overlay(stroke).shadow(…)`).
- A11y des deux composants déjà conforme (labels localisés sur clear-search, confirmer,
  republier ; hint « toucher pour répondre » sur la bulle ; gestes dismiss tap/drag de
  `StatusBubbleOverlay` intacts). Non touchée.
- Les deux fichiers `import MeeshyUI` (atome accessible). Aucun test n'asserte le fond
  matériau de ces vues (grep `LocationPicker`/`StatusBubbleOverlay` dans Tests = 0).
