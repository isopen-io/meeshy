# Analyse UI/UX — Itération 91i (2026-07-01) — iOS

## Écran audité
`apps/ios/Meeshy/Features/Main/Views/NewConversationView.swift` — écran de création de
conversation (recherche d'utilisateurs, sélection multiple → groupe, saisie du nom de groupe).
Écran à fort trafic (point d'entrée de toute nouvelle conversation).

## Contexte
Piste iOS indépendante (suffixe `i`). Poursuite du sweep Dynamic Type + VoiceOver amorcé aux
itérations 55i → 90i. La vue avait **17 tailles de police figées** (`.font(.system(size:))`),
donc aucun texte ne suivait le réglage Dynamic Type de l'utilisateur — problème d'accessibilité
identique à ceux déjà soldés sur `DataExportView` (90i), `MagicLinkView` (90i), `DeleteAccountView`
(88i), `EffectsPickerView` (89i).

## Constats

### 1. Typographie non-scalable (17 sites)
Titres, boutons, noms d'utilisateur, placeholders de champs, libellé « Bloqué », état vide :
tous en taille fixe. Au réglage Dynamic Type XL/AX, le texte restait figé → illisible pour les
utilisateurs malvoyants.

### 2. Boutons VoiceOver sans label (2 sites)
- Bouton `xmark.circle.fill` de retrait d'un utilisateur sélectionné (chip) → VoiceOver
  annonçait « xmark.circle.fill » (nom brut du symbole SF).
- Bouton `xmark.circle.fill` d'effacement de la recherche → même problème.

### 3. Glyphes décoratifs polluant VoiceOver (5 sites)
`person.3.fill`, `magnifyingglass`, `person.slash` (empty-state), `hand.raised.fill` (indicateur
bloqué, doublonné par le texte « Bloqué »), `checkmark.circle`/`circle` (état de sélection,
redondant avec l'action de la row) — tous lus inutilement par VoiceOver.

### 4. État de sélection non exposé sémantiquement
La row utilisateur (`Button`) signalait la sélection **uniquement** par un glyphe visuel
(checkmark) → VoiceOver ne communiquait pas l'état sélectionné/non-sélectionné.

## Corrections appliquées
- **16/17** `.font(.system(size:))` → `MeeshyFont.relative(size, weight:)` (weight préservé).
- **1 site gardé FIXE & commenté** : `chevron.left` 16pt du header (chrome nav glyph, doctrine
  82i/87i/90i — les chevrons/xmark de toolbar/header ne scalent pas).
- **Labels VoiceOver** ajoutés sur les 2 boutons `xmark.circle.fill` :
  retrait de chip (`accessibility.remove_selected_user`, **clé neuve**, 5 langues) +
  clear recherche (`accessibility.clear_search`, **clé existante réutilisée**).
- **5 glyphes décoratifs** masqués (`.accessibilityHidden(true)`).
- **Empty-state** groupé (`.accessibilityElement(children: .combine)`) → une seule annonce.
- **Trait `.isSelected`** ajouté sur la row sélectionnée (`.accessibilityAddTraits`).

## Invariants respectés
- 1 fichier de vue touché + 1 entrée xcstrings ; 0 changement de logique ; 0 test neuf ;
  layout/palette par défaut inchangés ; SDK non touché.
- Dérogation assumée à « 0 clé i18n neuve » : 1 clé ajoutée pour un contrôle interactif
  auparavant **sans** label VoiceOver (gain a11y réel, pas cosmétique).

## Gate
CI `ios-tests.yml` (compile Xcode 26.1 + tests simu iOS 18.2). SwiftUI ne compile pas sous Linux
→ CI seule autorité.

## Statut
✅ **SOLDÉ 91i** — `NewConversationView` : Dynamic Type + VoiceOver complets. **NE PAS re-flagger**
la typographie/VoiceOver de cet écran ni son `chevron.left` 16pt figé (décision doctrine chrome).
