# Analyse UI/UX — Itération 91i (2026-07-01) — iOS

## Écran audité
`apps/ios/Meeshy/Features/Main/Views/NewConversationView.swift` — écran de création de
conversation (recherche d'utilisateurs, sélection multiple → groupe, saisie du nom de groupe).
Écran à fort trafic (point d'entrée de toute nouvelle conversation).

## Contexte
Piste iOS indépendante (suffixe `i`). Poursuite du sweep Dynamic Type + VoiceOver amorcé aux
itérations 55i → 90i. La vue avait **17 tailles de police figées** (`.font(.system(size:))`),
donc aucun texte ne suivait le réglage Dynamic Type de l'utilisateur — problème d'accessibilité
identique à ceux déjà soldés sur `DataExportView` (90i), `MagicLinkView` (90i), `DeleteAccountView`
(88i), `EffectsPickerView` (89i).

## Constats

### 1. Typographie non-scalable (17 sites)
Titres, boutons, noms d'utilisateur, placeholders de champs, libellé « Bloqué », état vide :
tous en taille fixe. Au réglage Dynamic Type XL/AX, le texte restait figé → illisible pour les
utilisateurs malvoyants.

### 2. Boutons VoiceOver sans label (2 sites)
- Bouton `xmark.circle.fill` de retrait d'un utilisateur sélectionné (chip) → VoiceOver
  annonçait « xmark.circle.fill » (nom brut du symbole SF).
- Bouton `xmark.circle.fill` d'effacement de la recherche → même problème.

### 3. Glyphes décoratifs polluant VoiceOver (5 sites)
`person.3.fill`, `magnifyingglass`, `person.slash` (empty-state), `hand.raised.fill` (indicateur
bloqué, doublonné par le texte « Bloqué »), `checkmark.circle`/`circle` (état de sélection,
redondant avec l'action de la row) — tous lus inutilement par VoiceOver.

### 4. État de sélection non exposé sémantiquement
La row utilisateur (`Button`) signalait la sélection **uniquement** par un glyphe visuel
(checkmark) → VoiceOver ne communiquait pas l'état sélectionné/non-sélectionné.

## Corrections appliquées
- **16/17** `.font(.system(size:))` → `MeeshyFont.relative(size, weight:)` (weight préservé).
- **1 site gardé FIXE & commenté** : `chevron.left` 16pt du header (chrome nav glyph, doctrine
  82i/87i/90i — les chevrons/xmark de toolbar/header ne scalent pas).
- **Labels VoiceOver** ajoutés sur les 2 boutons `xmark.circle.fill` :
  retrait de chip (`accessibility.remove_selected_user`, **clé neuve**, 5 langues) +
  clear recherche (`accessibility.clear_search`, **clé existante réutilisée**).
- **5 glyphes décoratifs** masqués (`.accessibilityHidden(true)`).
- **Empty-state** groupé (`.accessibilityElement(children: .combine)`) → une seule annonce.
- **Trait `.isSelected`** ajouté sur la row sélectionnée (`.accessibilityAddTraits`).

## Invariants respectés
- 1 fichier de vue touché + 1 entrée xcstrings ; 0 changement de logique ; 0 test neuf ;
  layout/palette par défaut inchangés ; SDK non touché.
- Dérogation assumée à « 0 clé i18n neuve » : 1 clé ajoutée pour un contrôle interactif
  auparavant **sans** label VoiceOver (gain a11y réel, pas cosmétique).

## Gate
CI `ios-tests.yml` (compile Xcode 26.1 + tests simu iOS 18.2). SwiftUI ne compile pas sous Linux
→ CI seule autorité.

## Statut
✅ **SOLDÉ 91i** — `NewConversationView` : Dynamic Type + VoiceOver complets. **NE PAS re-flagger**
la typographie/VoiceOver de cet écran ni son `chevron.left` 16pt figé (décision doctrine chrome).

## Prochaines cibles (92i+)
`AffiliateView` (17), `LocationPickerView` (17), `MemberManagementSection` (17) ; puis
`StoryViewerView+Content` (31) et `ConversationView+Composer` (22) en dernier ; Glass adoption
`MessageOverlayMenu` (21).

## État avant

- **15** sites `.font(.system(size:))` → figent la taille, **ignorent totalement Dynamic Type**.
- Le rotor VoiceOver « En-têtes » ne captait ni le titre d'écran ni l'en-tête de section « MES COMMUNAUTÉS » (pas de trait `.isHeader`).

### Dynamic Type — 13/15 sites migrés
`.font(.system(size: N, weight:))` → `MeeshyFont.relative(N, weight:)` (weight préservé). Couvre : chevron retour, titre d'écran, bouton créer (`plus.circle.fill`), icône + valeur + libellé des 3 cartes de stats, en-tête de section, titre + sous-titre de l'état vide, nom + sous-titre de chaque ligne, glyphe du bouton copier (`doc.on.doc`), `chevron.right` de disclosure.

### 2 sites gardés FIXES (justifiés, commentés in-situ)
1. `person.3.fill` **40pt** héros de l'état vide — décoratif ≥40pt, le libellé adjacent porte le sens (doctrine 74i/86i). **`.accessibilityHidden(true)` ajouté.**
2. `person.3.fill` **14pt** dans un cercle de dimension fixe **40×40** de la ligne — figé (déborderait s'il scalait, doctrine 86i). **`.accessibilityHidden(true)` ajouté.**

### VoiceOver — traits déclaratifs
- `.accessibilityAddTraits(.isHeader)` ×2 : titre d'écran + en-tête de section (rotor « En-têtes », parité `AboutView` 86i / `DataExportView` 90i).
- `.accessibilityElement(children: .combine)` sur `communityStatCard` (lit « 12 Groupes » d'un bloc) + sur le VStack de l'état vide (titre + sous-titre lus ensemble).
- `.accessibilityHidden(true)` sur 4 glyphes décoratifs (icône de carte, 2 héros/glyphes fixes ci-dessus, `chevron.right`).

## Hors-scope (documenté, NON changé)
- La **ligne** (`communityLinkRow`) n'est **pas** combinée en un seul élément VoiceOver : elle contient un bouton « Copier le lien » imbriqué qui doit rester atteignable — seuls les glyphes décoratifs sont masqués, le `NavigationLink` et le bouton copier gardent leurs libellés.
- Le bouton copier utilise `UIPasteboard.general.string` direct (pas la source unique `copyToClipboard` évoquée sur la piste web) — convergence hors périmètre de ce sweep typographique iOS.

## Vérification
- Sweep présentation + traits a11y déclaratifs, **0 logique modifiée, 0 clé i18n neuve, 0 test neuf** (parité 55i/71i/74i/83i/86i/87i/88i/90i).
- Compte : 13 `relative` + 2 `.system(size:)` figés = 15 sites d'origine. ✅
- Tailles de base inchangées à la cadence Dynamic Type par défaut (`.large`) → **zéro régression visuelle** au réglage standard.
- `MeeshyFont.relative` résout sans `import MeeshyUI` (précédent prouvé : `MessageInfoSheet.swift` mergé utilise le même symbole avec le même jeu d'imports SwiftUI/Combine/MeeshySDK).
- Gate = CI `iOS Tests` (compile Xcode 26.1.x + simulateur 18.2). Environnement de dev Linux → build iOS non exécutable localement, CI seule autorité.

## Annotation continuité (à ne plus re-flagger)
- ⚠️ **Typographie/VoiceOver de `CommunityLinksView` = SOLDÉ 91i.** Les 2 `.system(size:)` restants (héros état-vide 40 + glyphe ligne 14 dans cercle fixe 40×40) sont figés à dessein + masqués VoiceOver.
- Sibling `CommunityLinkDetailView.swift` (10 sites `.system(size:)`, même import set) reste **candidat Dynamic Type ouvert** pour une itération future.

## Diagnostic

### Dynamic Type — 17 sites `.font(.system(size:))` figés
L'écran entier ignorait Dynamic Type : header, cartes de stats, section « MES LIENS », état vide, et rangées de tokens. Aucun texte ne scalait avec le réglage d'accessibilité « Larger Text ».

### Palette — hors-scope (documenté)

- Héros décoratif `link` **36pt** de l'état vide (`emptyTokensState`) : le libellé adjacent porte le sens → gardé `.system(size: 36)` + `.accessibilityHidden(true)` (parité doctrine 74i/86i : gros glyphe ornemental dont la sémantique est dans le texte adjacent reste figé et masqué de VoiceOver plutôt que de scaler et déséquilibrer l'état vide).

### VoiceOver — labels & traits (0 clé i18n neuve)
