# UI/UX Analysis — Iteration 72i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **Dynamic Type / accessibilité (4y11) de la feuille de commentaires du fil**
(`FeedCommentsSheet.swift`). Itération bornée à **un seul fichier de production**, dans la
continuité directe du différé « Dynamic Type grandes surfaces » listé par 53i/54i/55i/71i
(`FeedCommentsSheet` = 28 tailles de police figées). C'est l'une des surfaces les plus riches
en **texte de lecture** (corps des commentaires, réponses, noms d'auteur) — donc l'une de celles
où l'absence de Dynamic Type pénalise le plus la lisibilité.

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de
build de validation (SwiftUI ne compile pas sous Linux). Ce lot est un **sweep typographique
mécanique** sans logique pure isolable — pas de nouveau test unitaire (parité avec 55i
`ConversationInfoSheet` et 71i `TwoFactorSetupView`, deux sweeps Dynamic Type purs) ; le gate
est la compile CI + la suite existante (`CommentsSheetView`/`CommentRowView` ont déjà leurs
tests de comportement, inchangés).

## Contexte / point de départ
- `apps/ios/Meeshy/Features/Main/Views/FeedCommentsSheet.swift` (1629 lignes) héberge la feuille
  de commentaires (`CommentsSheetView`), la ligne de commentaire (`CommentRowView`), la section
  threadée (`ThreadedCommentSection`) et l'aperçu de post. **28** call-sites figeaient la taille
  de police via `.font(.system(size:))`.
- Conséquence a11y : un utilisateur ayant agrandi la police système (Réglages › Accessibilité ›
  Texte plus grand) lisait les **commentaires** (corps, réponses, noms) à taille **fixe**. Or
  lire des commentaires/réponses est exactement l'usage où le Dynamic Type compte.

## iOS Findings

### Dynamic Type — `.system(size:)` figé → `MeeshyFont.relative(...)` scalable (texte + glyphes accolés)
**23 conversions** vers `MeeshyFont.relative(size, weight:, design:)`
(`packages/MeeshySDK/Sources/MeeshyUI/Theme/Accessibility.swift` — mappe la taille legacy vers
le `Font.TextStyle` relatif le plus proche, weight/design préservés). Le swap est **mécanique** :
`.font(.system(size: X, …))` → `.font(MeeshyFont.relative(X, …))`.

- **Tous les `Text`** : titre de barre (`feed.comments.count`), aperçu de post (auteur, temps,
  contenu, compteurs likes/commentaires), bannière de réponse (auteur + extrait), **corps du
  commentaire** (`contentFont` 14/15 — le contenu de lecture central), **nom d'auteur**
  (`authorFont` 13/14), séparateurs `·`, horodatage relatif, compteur de likes, compteur/libellé
  « Répondre », « Voir ». Les tailles variables (`authorFont`/`contentFont`, computed `CGFloat`)
  et ternaires (`isReply ? 12 : 14`) se convertissent proprement (`relative(_:)` prend un `CGFloat`).
- **Glyphes inline accolés à un libellé dans le MÊME HStack** (donc qui doivent scaler avec lui) :
  `heart.fill`/`bubble.right.fill` des stats de l'aperçu de post (accolés à leur compteur),
  `heart`/`heart.fill` du bouton « J'aime » (accolé au compteur), `arrowshape.turn.up.left` du
  bouton « Répondre » (accolé au libellé), `ellipsis` du menu « … » (même rangée d'actions que
  cœur + réponse — converti pour rester aligné avec les libellés de la rangée qui scalent).
- Gain : le corps des commentaires, les réponses et toute la méta scalent désormais avec la
  préférence Dynamic Type, sur les commentaires racine **et** les réponses (`isReply`).

### Périmètre délibérément exclu (ne pas re-flagger) — 5 call-sites gardés figés
- **Bouton « Fermer » de la barre** (`xmark` 14, ligne 373) : **chrome de navigation** dans une
  pastille circulaire **à frame fixe `32×32`**. Le glyphe ne doit pas grossir au point de toucher
  le cercle ; les contrôles de barre ne scalent pas (parité HIG / doctrine 71i sur le chrome).
- **Bouton « Annuler la réponse »** (`xmark` 10, ligne 763) : idem — chrome dans une pastille
  **à frame fixe `24×24`**.
- **Toggles de drapeau de langue** (`isOrigActive ? 12 : 10` / `isTransActive ? 12 : 10`, lignes
  1372/1399) : contrôles de bascule **à layout précis** — le drapeau est surmonté d'un soulignement
  actif **à largeur fixe `10×1.5`**. Faire scaler le drapeau sans son soulignement casserait
  l'alignement du chip de bascule. Gardés figés (contrôle précis, pas du texte de lecture).
- **Icône `translate` décorative** (10, ligne 1421) : `.accessibilityHidden(true)`, accolée au
  cluster de drapeaux ci-dessus → gardée figée avec lui pour cohérence visuelle du chip.

## Conforme (vérifié — ne pas re-flagger)
- **Copie de contenu** : le menu « … » expose déjà « Copier le texte »
  (`UIPasteboard.general.string`, gaté par `canCopyContent` — absent sur un commentaire
  média-seul). Conforme — geste de copie naturel présent.
- **Gestes de feuille** : `presentationDetents([.large, .medium])` + `presentationDragIndicator`
  (swipe-to-dismiss natif) + bouton fermer. Conforme.
- **Cibles tactiles** : `.frame(minHeight: 44)` / `.meeshyTapTarget(44)` sur like / répondre /
  voir / fermer / annuler / plus. Conforme (≥ 44 pt).
- **Couleurs** : tokens charte (`MeeshyColors.error`/`.indigo400`, `theme.text*`,
  `Color(hex: accentColor)` = couleur d'accent de conversation), pas de hex sémantique dur.
  Dark/light gérés via `ThemeManager` + `@Environment(\.colorScheme)`. Conforme.
- **a11y labels/traits** : auteur (`.isButton` + label/hint profil), corps (label horodatage +
  contenu), like (label/value/hint), drapeaux (label/value langue), « … » labellisé. Conforme.

## Déblocage CI (hors-périmètre, requis pour merger — récidive du fix 36df5cb2)
La compile iOS de `main` était **de nouveau cassée** au moment du run CI de cette PR :
`ReplyThreadOverlay.swift:98` référence `ReplyThreadLoader`, type **supprimé** du codebase →
`error: cannot find 'ReplyThreadLoader' in scope` → `** TEST FAILED **` (exit 65 = échec de
**compile**). **Toute PR iOS était rouge**, dont celle-ci.

Cause : `ReplyThreadOverlay.swift` est une **vue morte** (jamais instanciée, remplacée par
`ThreadView`), supprimée avec sa dépendance `ReplyThreadLoader.swift` en `7d26cd91`. Un merge
ultérieur (`84fedd79`, une PR gateway) l'a **re-ressuscitée** via artefact de merge — récidive
exacte du problème que `36df5cb2` avait corrigé. Fix : **compléter la suppression** (zéro
référence externe vérifiée ; `ReplyThreadLoader` n'est référencé que depuis ce fichier). CI
régénère le projet via XcodeGen (globbing disque) → le fichier supprimé sort du build
automatiquement ; `project.pbxproj` (artefact) non édité à la main. Identique au 2ᵉ commit de
#1137 (71i) — les deux suppressions sont idempotentes au merge.
