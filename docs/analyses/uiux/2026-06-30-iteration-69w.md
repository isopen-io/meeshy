# Analyse UI/UX — Itération 69w (Web)

> **Scope** : `apps/web` **exclusivement**. Les vues iOS ne servent que de référence de parité (couleurs/features naturelles Meeshy), jamais d'objet de revue.
> **Thème** : accessibilité clavier (WCAG 2.1.1 *Keyboard* / 4.1.2 *Name, Role, Value* / 2.4.7 *Focus Visible*) du **modal de création de lien de conversation** — en-têtes de section repliables + substitut de case à cocher, tous `<div>`/`CardHeader` souris-only. Catégorie « **différé prioritaire 69w+** » du pointeur autoritaire 68w (a11y clavier des `<div onClick>` non-`<button>` HORS video-calls/conversation-list).

## Contexte
La routine a déjà soldé exhaustivement : anti-pattern i18n `t('key')||'fallback'`, aria-labels de contenu, focus-traps modales, `prefers-reduced-motion` global (MotionProvider #862), consolidation tokens dark-mode, épuration code mort, lazy-load images, **a11y clavier liste de conversations + tuile audio (67w / #1078)** et **contrôles plein écran d'appel vidéo (68w / #1082)** — tous deux mergés sur `main`.

Le différé prioritaire 68w+ visait l'audit transverse des `<div onClick>`/`role="button"` sans `onKeyDown`/`focus-visible`. Un audit ciblé de `apps/web` (hors clusters déjà soldés) a remonté 13 candidats ; le cluster le plus **cohérent, user-facing et auto-contenu** est le **modal de création de lien** (`create-link-modal/`), où l'intégralité des contrôles de configuration est inaccessible au clavier.

## Constats vérifiés (file:line) et corrections

| # | Fichier | Problème | Correction |
|---|---------|----------|-----------|
| 1 | `components/conversations/create-link-modal/steps/config-sections/LanguagesSection.tsx:32-35` | En-tête repliable « Langues autorisées » = `CardHeader` avec `onClick` toggle **souris-only** : pas de `role`, `tabIndex`, `onKeyDown`, ni état `aria-expanded`. Inactivable/invisible au clavier et au lecteur d'écran. | `role="button"` + `tabIndex={0}` + `aria-expanded={isLanguagesOpen}` + `onKeyDown` Enter/Space (`preventDefault`) + anneau `focus-visible:ring-inset`. Le nom accessible vient du `CardTitle` contenu (clé i18n existante). |
| 2 | `components/conversations/create-link-modal/steps/config-sections/PermissionsSection.tsx:66-69` | En-tête repliable « Permissions » = même `CardHeader` toggle **souris-only**, mêmes manques. | Idem #1 avec `aria-expanded={isPermissionsOpen}`. |
| 3 | `components/conversations/create-link-modal/components/SelectableSquare.tsx:15-26` | **Substitut de case à cocher réutilisé pour TOUS les toggles du modal** (langues, messages anonymes, images, fichiers, historique, compte requis, pseudo, e-mail, anniversaire). `<div onClick>` **souris-only** : pas de `role`, `tabIndex`, `onKeyDown`, ni `aria-checked`/`aria-disabled`. C'est le contrôle réel que l'utilisateur bascule — gap **plus impactant** que les en-têtes (un en-tête sans sa case accessible serait incohérent). | `role="checkbox"` + `aria-checked={checked}` + `aria-label={label}` + `aria-disabled={disabled}` + `tabIndex={disabled ? -1 : 0}` (retrait du tab order si désactivé) + `onKeyDown` Enter/Space (no-op si `disabled`, `preventDefault`) + anneau `focus-visible:ring-offset-1`. Clic souris inchangé. |

> **Pourquoi ce cluster est cohérent** : les 3 contrôles vivent dans le même flux (`create-link-modal/`), partagent l'action « basculer un état de configuration » et le même défaut clavier. Corriger les en-têtes sans les cases (ou l'inverse) laisserait le modal partiellement inutilisable au clavier. **Aucune nouvelle clé i18n** (les libellés/titres existants servent de nom accessible).
