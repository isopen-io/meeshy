# Analyse UI/UX — Itération 69w (Web)

> **Scope** : `apps/web` **exclusivement**. Les vues iOS ne servent que de référence de parité (couleurs/features naturelles Meeshy), jamais d'objet de revue.
> **Thème** : accessibilité clavier (WCAG 2.1.1 *Keyboard* / 2.4.7 *Focus Visible*) des **previews d'attachments en zone de réponse** — `<div role="button">` Tab-focusables mais **non activables au clavier** (pas de `onKeyDown`) et **sans anneau de focus visible**.

## Contexte
La catégorie « **a11y clavier des `<div onClick>` non-`<button>`** » a été ouverte par 67w (liste de conversations + tuile audio, #1078) puis 68w (contrôles plein écran appel vidéo, #1082, mergé). Le pointeur 68w signalait : « Reste à balayer hors video-calls/conversations pour itérations futures (audit transverse `role="button"` sans `onKeyDown`/`focus-visible`) ».

Audit transverse ciblé (`<div ... onClick>` dans `components/`, hors surfaces en vol #1084 create-link-modal / #1077 auth) → cluster cohérent confirmé dans **`components/attachments/AttachmentPreviewReply.tsx`**.

> **Anti-doublon vérifié** : `ImageAttachment.tsx` / `FileAttachment.tsx` (même répertoire) ont **déjà** `role`/`tabIndex`/`onKeyDown`/`aria-label`/`focus-visible` (soldés antérieurement). `MediaImageCard.tsx` utilise déjà des `<button>` natifs. `MediaVideoCard.tsx:245` est un **backdrop** click-outside (pas de besoin clavier). Le cluster `AttachmentPreviewReply` est le seul gisement net restant sur cette surface.

### Implémentation — épuration / DRY
- Helper unique **`activateOnEnterOrSpace`** (mémoïsé `useCallback`) : `(action) => (e) => { if (Enter|Espace) { preventDefault(); action(e); } }`. Un seul point de vérité pour les 3 contrôles, pas de répétition inline.
- Les 3 handlers existants (`handleImageClick`/`handlePdfClick`/`handleTextClick`) voient leur signature d'événement **élargie** `React.MouseEvent → React.MouseEvent | React.KeyboardEvent` (les deux exposent `stopPropagation`), ce qui **évite tout cast `as unknown`** (contrairement au pattern hérité de `ImageAttachment.tsx`). Type-safe, aligné sur les guidelines TS du repo (« No type assertions without justification »).
- Clés i18n : **0 clé neuve** — les `aria-label` (`actions.openImageNamed` / `actions.openPdfNamed` / `actions.openTextFileNamed`) existaient déjà ×4 locales (en/fr/es/pt).
