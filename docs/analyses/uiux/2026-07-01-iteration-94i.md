# Itération 94i — Analyse UI/UX iOS

**Date** : 2026-07-01
**Piste** : iOS uniquement (suffixe `i`)
**Surface** : `apps/ios/Meeshy/Features/Main/Views/SharePickerView.swift`
**Écran** : « Partager avec... » — feuille de partage de contenu (texte / lien / image / message transféré / story) vers une conversation.

## Contexte / anti-collision

Run 2026-07-01 : très forte contention iOS (~10 PRs ouvertes, labels 91i→93i) sur des surfaces
**disjointes** — `AffiliateView` (#1233/1234/1235/1238), `NewConversationView` (#1237),
`CommunityLinksView` (#1236), `DataExportView` (#1226 mergé), `LocationPickerView` (#1240/1242 + 93i mergé),
`ConversationPreferencesTab` (#1241), plus PRs web. **`SharePickerView` n'est prise par aucune PR.**
Numéro **94i** = > 93i (plus haut label en vol) pour éviter toute collision.

Thématiquement alignée avec le brief : « sélection de contenu, copie de contenu » — c'est l'écran
qui reçoit tout contenu partagé (`SharedContentType`) et le route vers une conversation.

## Constat initial

`SharePickerView` avait **15 sites `.font(.system(size:))`** figés → texte insensible au réglage
Dynamic Type. VoiceOver : la bannière d'aperçu et les rangées de conversation se lisaient en
fragments (icône + label + preview séparés) ; le glyphe d'état « envoyé » (checkmark) n'avait
**aucun** label.

Palette : **déjà entièrement tokenisée** (`MeeshyColors.indigo400/500/600`, `.warning`, `.success`,
`theme.*`) → 0 swap. Les fonds de surface neutres (`Color.white/black.opacity(0.02–0.03)`) sont des
surfaces d'aperçu, hors-scope (documenté).

Glass : la feuille suit le style de navigation natif (`NavigationStack` + `.navigationTitle`), pas
de surface flottante → pas d'adoption Glass pertinente.

## Décisions

### Dynamic Type — 13/15 migrés
`.font(.system(size: N[, weight:]))` → `MeeshyFont.relative(N[, weight:])` (weight préservé) sur :
- Bannière d'aperçu : `contentLabel` (11 semibold), `contentPreview` (12), les **4** glyphes de
  type de contenu (`text.bubble.fill`/`link.circle.fill`/`arrowshape…`/`play.rectangle.fill`, 16) —
  appariés à un texte scalable → migrés pour scaler ensemble.
- Champ de recherche : loupe (14 medium), `TextField` (15), bouton clear `xmark.circle.fill` (16).
- Rangée de conversation : titre (`ConversationTitleLabel.font` accepte un `Font`, 15 medium),
  libellé de type (12), puce séparatrice `•` (10), aperçu du dernier message (12).

### 2/15 gardés FIXES & commentés
Les glyphes du **slot d'action de rangée** — `checkmark.circle.fill` (état envoyé) et
`paperplane.circle.fill` (bouton envoyer), tous deux **26pt** — restent `.system(size: 26)` :
taille de contrôle délibérée occupant la zone de tap trailing (≈44pt), pas du texte de lecture
(doctrine 74i/86i). Chacun commenté in-situ.

### VoiceOver
- `.accessibilityElement(children: .combine)` sur la **bannière d'aperçu** (lit « Texte, <preview> »
  en un élément) et sur le **bloc texte de chaque rangée** (lit « <nom>, Groupe • <preview> »).
- `.accessibilityHidden(true)` sur les 3 glyphes décoratifs appariés à du texte : icône de type de
  contenu (bannière combinée), loupe de recherche, puce séparatrice `•`.
- `.accessibilityLabel(share.sent)` sur le checkmark d'état « envoyé » — **comble un trou a11y réel**
  (le glyphe ne portait aucun label). Bouton envoyer et bouton clear avaient déjà leur label.

### i18n
1 clé neuve **`share.sent`** (de/en/es/fr/pt-BR) — insérée en texte brut dans `Localizable.xcstrings`
au style Xcode exact (pas de reformatage du catalogue : diff = +36/−1). Aucune autre chaîne modifiée.

## Portée / vérification
- 1 fichier de production + 1 entrée catalogue + docs routine.
- Compte : 13 `MeeshyFont.relative` + 2 `.system(size: 26)` figés = 15 d'origine ✅.
- **0 changement de logique, 0 test neuf** (sweep présentation + traits déclaratifs, parité
  55i/71i/74i/83i/86i/88i/90i/91i).
- Tailles de base inchangées à la cadence Dynamic Type par défaut (`.large`) → zéro régression
  visuelle au réglage standard.
- SwiftUI ne compile pas sous Linux → **gate = CI `iOS Tests`** (`ios-tests.yml` : xcodegen →
  build-for-testing Xcode 26.1 → tests simu iOS 18.2).

## Solde / différé
- **`SharePickerView` SOLDÉ** : Dynamic Type + VoiceOver + palette. **NE PLUS re-flagger** ; les 2
  `.system(size: 26)` (glyphes de contrôle du slot d'action) sont figés à dessein.
- **Différé prioritaire iOS 95i+** : `MemberManagementSection` (17), `AddParticipantSheet` (14),
  `ForwardPickerSheet` (9, jumeau du présent écran), `ConversationView+Composer` (22, lot critique
  prudent), `StoryViewerView+Content` (38, ⚠️ collision i18n #1174) ; Glass adoption `MessageOverlayMenu`
  (21, via `AdaptiveGlassContainer`, lot dédié).
# Itération 94i — Analyse UI/UX iOS : `AffiliateView` + nettoyage merge `DataExportView`

**Date** : 2026-07-01
**Piste** : iOS (suffixe `i`) — indépendante des pistes Web/Android.
**Fichiers** :
- `apps/ios/Meeshy/Features/Main/Views/AffiliateView.swift` (écran Parrainage — cible principale)
- `apps/ios/Meeshy/Features/Main/Views/DataExportView.swift` (correction d'artefacts de merge)

## Contexte routine & choix du numéro
Le resync sur `main` a révélé une **contention iOS parallèle intense** : plusieurs agents ont numéroté « 90i » des surfaces différentes (`NewConversationView`, `FeedCommentsSheet`, `DataExportView`), et un « 93i » (`LocationPickerView`) — tous mergés. Pour éviter toute collision, cette itération prend **94i** (> plus haut numéro observé).

Surfaces déjà soldées par des agents parallèles (NE PAS re-toucher) : `NewConversationView` (90i), `FeedCommentsSheet` (90i), `DataExportView` (90i/91i), `LocationPickerView` (93i), `MagicLinkView` (90i=mon 90i), `EffectsPickerView` (89i).

Cible retenue **`AffiliateView`** : 17 sites police figée, aucune PR/itération en vol, forte valeur a11y (boutons icône-only non étiquetés = **vrai bug VoiceOver**).

## Partie A — `AffiliateView` (écran Parrainage)

### État avant
- **17** sites `.font(.system(size:))` → ignorent Dynamic Type.
- **4 boutons icône-only SANS `.accessibilityLabel`** (violation directe de la règle a11y CLAUDE.md « Every Button/Image MUST have accessibilityLabel ») : bouton « + » créer (header), bouton copier le lien, bouton partager, bouton supprimer. VoiceOver les annonçait par leur nom de glyphe (« plus circle fill », « doc on doc », …).
- Glyphe héros d'état vide `link` 36pt non masqué à VoiceOver.
- Palette conforme (accent vert `2ECC71` déterministe de la feature, `MeeshyColors.success/error`, `theme.*`).

### Changements
**Dynamic Type — 16/17 sites** → `MeeshyFont.relative(...)` (weight + `design: .rounded` des valeurs de stat + en-tête préservés) : chevron retour, titre, bouton +, icônes/valeurs/libellés des 3 cartes de stat, en-tête de section, titre/sous-titre d'état vide, nom du token, `Label` clics/inscrits, icônes copier/partager/supprimer.

**1 site gardé FIXE & commenté** : glyphe héros d'état vide `link` 36pt (décoratif ≥36pt, doctrine 74i/86i) + `.accessibilityHidden(true)`.

**VoiceOver — 4 labels comblés + 4 traits** :
- `.accessibilityLabel` sur les 4 boutons icône-only : créer (`affiliate.create.title`), copier (`common.copy`), partager (`common.share`), supprimer (`common.delete`) — **clés SSOT réutilisées, 0 clé neuve**.
- `.accessibilityAddTraits(.isHeader)` sur le titre d'écran + l'en-tête de section « MES LIENS ».
- `.accessibilityElement(children: .combine)` sur chaque carte de stat (icône masquée + valeur + libellé lus en un élément, ex. « 3 Liens »).
- `.accessibilityHidden(true)` sur glyphes décoratifs (héros vide, icônes de stat, icône d'en-tête).

## Partie B — Correction d'artefacts de merge `DataExportView`
Le merge parallèle de deux « 90i/91i » sur `DataExportView` avait laissé des **doublons** (défauts réels) :
1. **Double `.accessibilityElement(children: .combine)`** sur `infoCard` (2 modificateurs empilés) → dédupliqué (1 conservé, appliqué après le chrome visuel).
2. **Double `.accessibilityElement(children: .combine)`** sur `errorBanner` → dédupliqué.
3. **Double bloc de commentaire** sur le glyphe de badge `toggleRow` → fusionné en un seul.
4. **`.isHeader` redondant** : présent à la fois sur le `Text` interne ET sur le conteneur `sectionHeader` (déjà `.combine` + `.isHeader`) → retiré l'inner, gardé le conteneur (lecture icône+titre en un seul élément header).

État final `DataExportView` : 3 `.combine` légitimes (infoCard, errorBanner, sectionHeader), 2 `.isHeader` (titre écran + sectionHeader), 1 seul commentaire de badge. Aucune régression fonctionnelle — nettoyage pur.

## Vérification
- Sweep présentation + traits/labels déclaratifs + nettoyage de doublons, **0 logique modifiée, 0 clé i18n neuve, 0 test neuf** (parité 55i/74i/86i/88i/90i/91i).
- Tailles de base inchangées à la cadence Dynamic Type par défaut → **zéro régression visuelle**.
- `MeeshyFont`/`MeeshyColors` (module MeeshyUI) déjà résolus dans `AffiliateView` (ré-export transitif via `import MeeshySDK`, `MeeshyColors.success` déjà utilisé) → pas d'import ajouté.
- Gate = CI `ios-tests.yml`. Env dev Linux → build iOS non exécutable localement.

## Annotation continuité (à ne plus re-flagger)
- ⚠️ **Typographie/VoiceOver de `AffiliateView` = SOLDÉ 94i.** Le `.system(size:)` restant (héros vide 36pt) est figé à dessein. Les 4 boutons icône-only sont désormais étiquetés.
- ⚠️ **`DataExportView` : artefacts de merge nettoyés 94i.** Ne pas ré-empiler de `.combine`/commentaires.
# Analyse UI/UX — Itération 94i (iOS)

**Date** : 2026-07-01
**Piste** : iOS uniquement (suffixe `i`) — indépendante des itérations web.
**Surface** : `apps/ios/Meeshy/Features/Main/Components/MemberManagementSection.swift`
**Type** : sweep Dynamic Type + VoiceOver (présentation + traits déclaratifs, 0 logique).

## Contexte / anti-collision

`list_pull_requests` (2026-07-01, run 94i) : ~10 PR iOS ouvertes labellisées 91i→93i sur des
surfaces **disjointes** — `AffiliateView` (×4), `NewConversationView`, `CommunityLinksView`,
`DataExportView`, `LocationPickerView` (×2), `ConversationPreferencesTab`. `main` HEAD porte
déjà `LocationPickerView` (93i, #1225 mergé). `MemberManagementSection` est **flaggée « différé
prioritaire iOS 91i+ »** par le pointeur autoritaire (17 sites) et **n'apparaît dans aucune PR
ouverte** → surface orthogonale, zéro conflit attendu. Numéro **94i** = > 93i (plus haut label
en vol).

`MemberManagementSection` = section de gestion des membres d'une conversation (barre de recherche,
liste des membres avec badges de rôle Creator/Admin/Modérateur, menu d'actions promouvoir /
rétrograder / expulser / bannir, bouton « Ajouter un membre », états loading/vide).

## État avant

- **17 sites `.font(.system(size:))`** figés → le texte de la section ignorait totalement le
  réglage Dynamic Type de l'utilisateur.
- Palette **déjà tokenisée** (`MeeshyColors.indigo600` teinte de section, `theme.*`,
  `MeeshyColors.info`/`.success` badges rôle, `DynamicColorGenerator.colorForName` avatars) → 0 swap.
- **1 lacune VoiceOver** : le bouton « ✕ » d'effacement de recherche n'avait aucun `.accessibilityLabel`.
- L'en-tête « MEMBRES » n'était pas exposé comme en-tête au rotor VoiceOver.

## Corrections appliquées (94i)

### Dynamic Type — 15/17 sites migrés
`.font(.system(size: N, weight:[, design:]))` → `MeeshyFont.relative(N, weight:[, design:])`
(weight + design `.rounded` préservés 1:1). Couvre :
- en-tête de section (glyphe `person.3.fill` + titre « MEMBRES (n) »),
- barre de recherche (loupe, `TextField`, bouton ✕),
- ligne membre : nom d'affichage,
- badges de rôle (3 branches : icône + libellé Creator / Admin / Modérateur),
- bouton « Ajouter un membre » (glyphe `person.badge.plus` + libellé),
- texte de l'état vide « Aucun membre trouvé ».

### 2/17 gardés FIXES & commentés
- **`ellipsis` 13pt** — glyphe chrome centré dans un cadre de tap fixe **32×32** ; une police
  scalable déborderait le cadre (doctrine 82i). Le `Menu` porte déjà un `.accessibilityLabel`
  (nom du membre via `member-management.options-a11y`).
- **`person.slash` 28pt** — glyphe hero décoratif de l'état vide ; le libellé adjacent porte le
  sens → gardé fixe + `.accessibilityHidden(true)` (précédent 90i `NewConversationView`).

### VoiceOver — 3 traits, comble 1 lacune réelle (0 clé i18n neuve)
- **`.accessibilityLabel(accessibility.clear_search)`** sur le bouton ✕ d'effacement de recherche
  (clé SSOT **existante**, réutilisée — lacune comblée).
- **`.accessibilityElement(children: .combine)` + `.accessibilityAddTraits(.isHeader)`** sur
  l'en-tête « MEMBRES » → navigable via le rotor « En-têtes » (convention 86i/89i), lu en un élément.
- **`.accessibilityHidden(true)`** sur les glyphes décoratifs appariés à un texte : `person.3.fill`
  (en-tête), loupe de recherche, 3 icônes de badge de rôle (`crown.fill`/`shield.fill`/
  `checkmark.shield.fill`), `person.badge.plus` (le bouton porte déjà son label), `person.slash`.
- **`.accessibilityElement(children: .combine)`** sur le bloc d'état vide (icône + texte lus en un).

## Hors-scope (documenté)

- **Palette** : déjà 100 % tokenisée → intacte. La seule couleur hex littérale restante,
  `Color(hex: "F8B500")` (or « Creator »), est une **teinte catégorielle de rôle** (échelle de
  rôles, pas un état sémantique) → laissée intacte conformément à la doctrine « échelle
  catégorielle » (69i/78i/89i).
- **i18n** : toutes les chaînes déjà `String(localized:)` ; le label a11y réutilise une clé
  existante → **0 clé neuve**.
- **Glass / layout** : inchangés (la section est un conteneur `RoundedRectangle` interne, pas une
  surface flottante candidate au Liquid Glass).

## Vérification

- Compte : **15 `relative` + 2 `.system(size:)` figés = 17** d'origine ✅
- **0 logique / 0 clé i18n neuve / 0 test neuf** (parité doctrine 55i/74i/86i/88i/89i/90i).
- `MeeshyFont.relative` est une extension **public** de `MeeshyFont` (enum public,
  `MeeshyUI/Theme/DesignTokens.swift` + `Accessibility.swift`), déjà importée transitivement
  via `import MeeshyUI` (le fichier consomme déjà `MeeshyColors`) → **aucun import ajouté**.
- Tailles de base à la cadence Dynamic Type `.large` inchangées → **zéro régression visuelle** au
  réglage standard.
- **Gate** = CI `iOS Tests` (SwiftUI ne compile pas sous Linux — compile Xcode 26.1 + tests simu 18.2).

## Statut

✅ **`MemberManagementSection` soldé (94i)** — typographie/VoiceOver conformes. Les 2 `.system(size:)`
restants (chrome `ellipsis` 32×32, hero `person.slash` 28pt) sont figés **à dessein** — **ne plus
re-flagger**.

## Différé prioritaire iOS 95i+
- Dynamic Type grandes surfaces restantes : `ConversationView+Composer` (22, lot dédié prudent =
  composer critique), `StoryViewerView+Content` (31, ⚠️ collision i18n historique #1174),
  `AboutView` (16), `CommunityLinkDetailView` (10).
- Glass adoption reste : `MessageOverlayMenu` (21, via `AdaptiveGlassContainer`, lot dédié).
