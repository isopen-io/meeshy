# UI/UX Analysis — Iteration 77i (2026-07-01)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android couverts par d'autres agents).
Thème : **i18n complète de `SharePickerView`** — la feuille « Partager avec… » présentait
une dizaine de chaînes **français hardcodées** visibles par tous les utilisateurs, quelle que
soit leur langue. Itération bornée à **un seul fichier de production**
(`SharePickerView.swift`) + le catalogue de localisation (`Localizable.xcstrings`).

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de
build de validation (SwiftUI ne compile pas sous Linux). Aucun test neuf : swap purement
mécanique de littéraux → clés de catalogue, aucune logique modifiée. Les clés réutilisées
(`conversation.type.*`, `common.close`) sont déjà exercées par les vues sœurs
(`GlobalSearchView`, `ConversationInfoSheet`).

## Contexte / point de départ
`SharePickerView` (`apps/ios/Meeshy/Features/Main/Views/SharePickerView.swift`) est la feuille
de partage (forward de message, partage de texte/lien/image/story vers une conversation).
Elle affichait **du français en dur** à l'utilisateur, non traduit :
- `navigationTitle("Partager avec...")`, `Button("Fermer")`
- `contentLabel` : "Texte", "Lien", "Image", "Message transféré", "Story partagée"
- `contentPreview` : "Photo à partager", "[Media]", "Story de \(authorName)"
- `TextField("Rechercher une conversation...")` (placeholder)
- `EmptyStateView(title: "Aucune conversation")`
- `conversationTypeLabel` : "Direct", "Groupe", "Publique", "Globale", "Communauté", "Canal",
  "Bot", "Communication"
- 2× `FeedbackToastManager.shared.showError("Erreur lors du partage")`
- payload sortant story : "🔗 Story de \(authorName) : <url>"

L'itération 3 (2026-06-08) avait **listé** ce fichier dans un survey des `accessibilityLabel`
français figés, mais **n'avait pas corrigé** les chaînes de chrome visibles ci-dessus.

## iOS Findings

### i18n — littéraux français → clés de catalogue (SSOT réutilisé quand dispo)
- **Réutilisation SSOT** : `conversationTypeLabel` migre vers la **même** famille de clés
  `conversation.type.{direct,group,public,global,community,channel,bot}` que
  `GlobalSearchView.conversationTypeLabel` (source de vérité) → un seul jeu de traductions
  pour les 8 types de conversation à travers l'app. `Button("Fermer")` → `common.close`
  (clé existante 5 langues).
- **Clé manquante ajoutée** : `conversation.type.broadcast` (jusque-là absente du catalogue →
  `GlobalSearchView` **et** `SharePickerView` tombaient sur le `defaultValue` français
  "Communication" en EN/ES/DE/pt-BR). Ajoutée avec traductions (en "Broadcast", es "Difusión",
  de "Broadcast", pt-BR "Transmissão", fr "Communication") → **corrige aussi GlobalSearchView**
  par effet SSOT.
- **Nouvelles clés `share.*`** (13) ajoutées ×5 langues : `share.picker.title`,
  `share.content.{text,url,image,message,story}`, `share.preview.{image,media,story}`,
  `share.search.placeholder`, `share.empty`, `share.error`, `share.story.shareText`.
- **Interpolation** : `share.preview.story` (`Story de %@`) et `share.story.shareText`
  (`🔗 Story de %1$@ : %2$@`) suivent le pattern établi
  `String(format: String(localized:defaultValue:bundle:), args…)` (cf. `ContactsListTab`,
  `RequestsTab`).

### Gain
- Parité linguistique : un utilisateur EN/ES/DE/pt-BR ne voit plus de français dans la feuille
  de partage ni dans les labels de type de conversation (y compris `GlobalSearchView` via SSOT).
- Source unique des labels de type (plus de duplication `direct/group/…` divergente entre
  `SharePickerView`, `GlobalSearchView`, `ConversationInfoSheet`).

## Conforme (vérifié — ne pas re-flagger)
- `contentPreview` `.text`/`.url` retournent le contenu brut (texte, URL) — données, pas chrome,
  ne pas localiser.
- `contentToSend` `.text`/`.url`/`.message` renvoient le contenu réel à envoyer — pas de chrome.
- Les `accessibilityLabel`/placeholders déjà localisés (lignes `common.clearSearch`,
  `share.sendTo`) — laissés tels quels.
- Couleurs (`MeeshyColors.indigo*`, `.success`, `.warning`) déjà tokenisées. Palette conforme.

## Anti-repetition check
Vérifié contre : itération 3 (survey a11y — a listé `SharePickerView` sans corriger le chrome),
70i (`ContactCardView` palette/a11y — fichier distinct), 68i (Glass GlobalSearch — a touché
`GlobalSearchView` mais pas ses `conversation.type.*`). Aucune itération n'avait localisé les
chaînes visibles de `SharePickerView` ni ajouté `conversation.type.broadcast`.

## Périmètre délibérément exclu (ne pas re-flagger)
- Les polices figées `.system(size:)` de `SharePickerView` (Dynamic Type) — relèvent du lot
  dédié « grandes surfaces polices figées », hors-scope de cette itération i18n.
- Le fond/`.ultraThinMaterial` éventuel — non concerné ici.

## Différés (pour itérations suivantes — iOS)
- Autres surfaces à littéraux français visibles (hors PRs en vol) : audit continu des
  `Text("…")`/`Button("…")`/`navigationTitle("…")` français figés composant par composant.
- Dynamic Type grandes surfaces (`CallView`, `StoryViewerView+Content`, `FeedView+Attachments`).
- Glass adoption (reste) : `MessageOverlayMenu` (lot dédié `AdaptiveGlassContainer`).
