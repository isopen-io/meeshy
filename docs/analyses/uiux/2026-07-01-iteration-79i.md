## Contexte / point de départ — surface user-visible + rupture Prisme
- `Route.displayTitle` (`apps/ios/Meeshy/Features/Main/Navigation/Router.swift:55`) renvoie le
  titre de chaque destination. Il alimente `Router.sceneTitle` (:162), lui-même propagé à
  **`UIWindowScene.title`** dans `RootView.swift:443` (`adaptiveOnChange(of: router.sceneTitle)`).
  → **Surface réellement visible** : titre de fenêtre en **iPadOS Stage Manager / multitâche**,
  **app switcher**, exposé à **VoiceOver** (annonce de scène). 23 des 25 branches du `switch`
  étaient des **littéraux FR figés** contournant toute localisation → titre de scène en français
  pour **tous** les utilisateurs (EN/ES/DE/pt-BR) = rupture du Prisme Linguistique.
- Défaut aggravant : la majorité de ces littéraux étaient **sans accents** (« Parametres »,
  « Decouvrir », « Communautes », « Parametres communaute », « Export de donnees ») — français
  dégradé même pour un utilisateur francophone.
- `Router.handleConversationDeepLink` (:316) affichait un toast d'échec en littéral FR brut
  (`FeedbackToastManager.shared.showError("Impossible d'ouvrir la conversation")`) — visible par
  **tous** les utilisateurs quand un deep-link `meeshy://` vers une conversation échoue.

## iOS Findings

### i18n — 25 titres de route + 1 toast deep-link → clés de catalogue ×5 langues
- **23 titres `displayTitle`** → `String(localized: "route.title.<x>", defaultValue: "<EN>", bundle: .main)`.
- **Fallback `sceneTitle`** (« Conversations », affiché quand `currentRoute == nil`) →
  `route.title.conversations`.
- **Toast deep-link** (:316) → `deeplink.conversation.error`.
- Les 26 clés sont **ajoutées au String Catalog** `Localizable.xcstrings`
  (`extractionState: manual`) avec traductions **×5 langues** (de/en/es/fr/pt-BR,
  `state: translated`). La valeur `fr` **corrige au passage les accents manquants**
  (« Parametres » → « Paramètres », « Decouvrir » → « Découvrir », « Communautes » →
  « Communautés », « Export de donnees » → « Export de données ») — la locale FR ne fait que
  **gagner ses accents** (amélioration, pas régression).

### Choix du `defaultValue` : anglais (anti « français-pour-tous »)
Les `defaultValue` en code sont en **anglais**, alignés sur le pattern déjà présent dans
`Router.swift` (`magicLink.success` = « Login successful! », :326 ; `magicLink.error.invalidLink`
= « Invalid or expired link », :329). Ainsi, même dans le cas improbable d'un catalogue absent,
le repli est un **anglais universel**, jamais du français forcé. Le catalogue portant les 5
langues (dont FR), le rendu par-langue reste correct au runtime.

### Namespace `route.title.*` neuf (plutôt que réutilisation d' SSOT dispersé)
La scène-titre est une **surface distincte** des en-têtes intra-écran (chaque écran a son
propre header localisé indépendamment). Des clés existantes sémantiquement proches existent
(`communities.title`, `settings.notifications.title`, `profile.friend_requests`,
`settings.tools.stats`…) mais appartiennent à des **contextes différents** (l'écran de réglages
de notifications ≠ le hub Notifications) — les réutiliser **coupleraient** le titre de scène à
des écrans sans rapport et risquerait une dérive sémantique. Un namespace `route.title.*`
**uniforme et auto-contenu** (une clé par `case`) est plus épuré et sans couplage.

## Conforme (vérifié — ne pas re-flagger)
- `Route.conversation` renvoie `conv.name` (donnée) — pas de littéral à traduire.
- `postDetail` renvoie `post?.author ?? <fallback>` — seul le **fallback** « Publication » est
  localisé ; le nom d'auteur reste de la donnée.
- Les toasts `magicLink.*` (:326/:329) sont **déjà** `String(localized:)` — laissés tels quels.
- Aucun changement de logique de navigation (`push`/`pop`/`handleDeepLink`), de `Route`
  `Hashable`, ni de comportement iPad two-column.
