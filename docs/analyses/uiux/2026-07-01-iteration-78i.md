# UI/UX Analysis — Iteration 78i (2026-07-01)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android couverts par d'autres agents).
Thème : **épuration palette — rouges destructifs / d'erreur / d'expiration codés en dur
→ token sémantique `MeeshyColors.error`**. Consolidation d'un cluster cohérent de couleurs
d'état négatif dispersées en `Color(hex:)` littéral à travers 7 fichiers (app + MeeshyUI).

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de
build de validation (SwiftUI ne compile pas sous Linux). **Aucun test neuf** : swap purement
littéral `Color(hex:)` → token, aucune logique modifiée. `MeeshyColors` est déjà importé/en
scope dans les 7 fichiers (app : `import MeeshyUI` présent ; MeeshyUI Media : même module,
3-4 refs `MeeshyColors.*` préexistantes par fichier).

## Contexte / point de départ
La charte Meeshy définit un token sémantique unique pour le rouge d'erreur :
`MeeshyColors.error = #F87171` (cf. `packages/MeeshySDK/Sources/MeeshyUI/Theme/MeeshyColors.swift`).
La table de migration couleur (CLAUDE.md app + SDK) mappe l'alias legacy `coral → error`.
Malgré cela, plusieurs indicateurs d'état négatif utilisaient encore un hex littéral :

- `#F87171` (**exactement** le token error, donc pixels identiques après swap) :
  - `StoryViewerView+Content.swift:1231` — fond capsule de la **bannière « Story expirée »**
    (icône `clock.badge.xmark`).
  - `NotificationRowView.swift:112` (MeeshyUI) — teinte de l'indicateur **contenu lié expiré**
    (`isLinkedContentExpired`, icône `clock.badge.xmark`).
- `#FF6B6B` (coral → error ; léger décalage de teinte, mais parité dark/light gagnée via token) :
  - `AddParticipantSheet.swift:167` — texte du **message d'erreur** (variable `errorMessage`).
  - `AudioPlayerView.swift:1571`, `CodeViewerView.swift:78`, `DocumentViewerView.swift:69` & `:144`,
    `ImageViewerView.swift:114` (MeeshyUI Media) — bouton **destructif de suppression**
    d'attachment (`Button { onDelete() } / xmark.circle.fill`) présent dans les 4 media viewers.

### Gain
- **Source unique du rouge d'erreur** : la suppression d'attachment est désormais visuellement
  identique dans les 4 media viewers (Audio/Code/Document/Image) — plus de dérive possible.
- **Parité dark/light** : `MeeshyColors.error` est le token tuné de la charte ; les littéraux
  `#FF6B6B` (coral non tuné) ne s'adaptaient pas.
- **Conformité charte** : applique la règle « nouveau code MUST use semantic names / coral→error »
  du CLAUDE.md (app + SDK).
- **Zéro risque de régression pixel** pour les 2 swaps `#F87171` (valeur identique au token) ;
  décalage volontaire et positif pour les `#FF6B6B` (aucun snapshot ne couvre ces vues —
  infra snapshot limitée à `MeeshyUITests/Timeline/Helpers`).

## Conforme (vérifié — NE PAS re-flagger)
- **`#FF6B6B` décoratifs / non-erreur laissés intacts** (sémantiques distinctes, hors-scope) :
  - `DynamicColorGenerator` type conversation `direct = #FF6B6B` (ladder catégoriel de couleur
    d'accent, cf. SDK CLAUDE.md — jamais un token sémantique).
  - `StoryFilterGridView.swift:87` / `StoryFilterPicker.swift:153` filtre `.vivid`
    (gradient décoratif).
  - `TagInputView.swift:78` palette de couleurs de tags (ladder décoratif).
  - `VoiceRecordingView.swift:103/117/119/179` rouge **enregistrement en cours**
    (`recorder.isRecording`) — sémantique « recording/live », **pas** erreur.
  - `LiveLocationBadge.swift:48/53` bouton « Arrêter » du partage de position — sémantique
    live-location dédiée, borderline destructif mais thématique → laissé.
  - `VoiceProfileManageView.swift:203` icône `trash` rouge — destructif mais dans le contexte
    voice-profile (famille recording-red) → laissé pour cohérence de fichier.
- Les autres hex figés recensés (`#4ADE80`, `#3B82F6`, `#9B59B6`, `#F8B500`, `#A855F7`…) ne
  correspondent PAS exactement à un token et/ou sont des ladders catégoriels → audit séparé
  (déjà différé par les itérations 69i/70i, décision « charte unique différée »).
