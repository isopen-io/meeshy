# UI/UX Analysis — Iteration 78i (2026-07-01)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **Dynamic Type / accessibilité de `LinkPreviewCard`** — la carte d'aperçu OpenGraph
rendue **sous chaque bulle de message contenant une URL** (et réutilisée en reply-preview,
starred-row). Itération bornée à **un seul fichier de production**
(`LinkPreviewCard.swift`), dans la continuité directe du différé « Dynamic Type grandes
surfaces / polices figées » listé par 55i+/69i/70i, mais sur une **petite surface fraîche**
non prise par les PR 71i–77i en vol.

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de
build de validation (SwiftUI ne compile pas sous Linux). `MeeshyFont.relative` est déjà
consommé par du code app (`ConversationInfoSheet` 55i, #1128) et couvert par les tests SDK
`Accessibility` — aucun test neuf requis pour un swap littéral→token.

## Contexte / point de départ
- `LinkPreviewCard` (`apps/ios/Meeshy/Features/Main/Components/LinkPreviewCard.swift`) affiche
  un aperçu OpenGraph (nom de site, titre, description, vignette) sous une bulle quand le texte
  contient une URL. Composant **totalement auto-porté** (résout ses métadonnées via
  `LinkPreviewStore.shared`, aucun couplage ViewModel), réutilisé en reply-preview et
  starred-message rows.
- Les 3 états (`populatedCard`, `skeletonCard`, `failedCard`) rendaient tout leur texte avec
  des **tailles de police figées** `.font(.system(size: 10/11/13/14))` — invisibles au réglage
  Dynamic Type / Larger Text de l'utilisateur. Un utilisateur malvoyant en « AX5 » voyait la
  carte figée à 10–13 pt.

## Pourquoi cette surface (et pas `ContactCardView`, écartée du Dynamic Type par 70i)
`ContactCardView` a été **exclu** du Dynamic Type par 70i à cause d'un `frame(width: 240)`
**dur** imposé par la bulle hôte (scaler casserait le layout borné). `LinkPreviewCard`, lui,
n'a **aucun cap de largeur** : seulement un plancher `frame(minHeight: 64)` (le contenu peut
grandir) + `lineLimit(1/2)` par champ + une vignette à taille fixe décorative
(`frame(width: 72, height: 72)`, correcte à garder figée). Le scaling est donc **sûr et
approprié** : le texte grandit et wrappe/tronque proprement, la carte grandit en hauteur.

## iOS Findings

### Dynamic Type — polices figées → styles relatifs scalables
Les **9** occurrences `.font(.system(size:))` de texte/glyphe → `MeeshyFont.relative(size:…)`
(swap mécanique documenté dans `Accessibility.swift`, poids/design préservés) :
- `populatedCard` : nom de site `10 semibold`, titre `13 semibold`, description `11`.
- `skeletonCard` : host `10 semibold`, url `11`.
- `failedCard` : host `10 semibold`, url `11`, glyphe `link` `14 semibold`.
- `thumbnail` placeholder : glyphe `link` `14 semibold`.

Gain : la carte respecte enfin le réglage Dynamic Type système (HIG a11y) ; texte lisible en
Larger Text au lieu d'un 10–13 pt figé. Zéro changement de couleur, de layout logique, ou de
comportement — pur swap `size:` → style relatif.

### Périmètre délibérément exclu (ne pas re-flagger)
- **Vignette `frame(width: 72, height: 72)` figée** : c'est une image OG décorative bornée,
  correcte à garder fixe (Dynamic Type concerne le texte, pas les images).
- **Barre d'accent 3 pt / rayons / paddings** : non liés au Dynamic Type.
- **`minHeight: 64` conservé** : plancher partagé skeleton→populated (évite le shift de bulle
  au chargement) — n'empêche pas la croissance.
- **Fond `MeeshyColors.indigo50/950` + border accent** : palette déjà tokenisée, conforme.

## Conforme (vérifié — ne pas re-flagger)
- Cible tactile : carte entière = `Button(.plain)` (≥ 44 pt via `minHeight: 64`). Conforme.
- `LinkPreviewStore.shared` en `let` (pas `@ObservedObject`) → « Zero Unnecessary Re-render »
  respecté (chaque carte pilote son état local par URL). Conforme.
- Couleur d'accent via `Color(hex: accentColor)` (couleur de conversation), pas un hex dur.

## Différés (pour 79i+)
- **Dynamic Type grandes surfaces restantes** (en vol / à confirmer) : `ConversationDashboardView`
  (#1145/#1165), `TwoFactorSetupView` (#1137/#1155), `InviteFriendsSheet` (#1160),
  `VoiceProfileManageView` (#1150), `FeedCommentsSheet` (#1139), `StoryViewerView+Content`,
  `FeedView+Attachments` — **vérifier `list_pull_requests` AVANT** (forte contention iOS).
- **Petites surfaces fraîches Dynamic Type** (mêmes critères que 78i, sans cap de largeur dur) :
  `AttachmentLoadingTile` (`.system(size: 8/10/16/20)`), `CameraView` (icônes `18/16/22`).
- **Palette sémantique fraîche** : `MemberManagementSection` (`#F8B500`→`warning`,
  `#3B82F6`→`info` ; le cas moderator utilise déjà `.success`), `AddParticipantSheet`
  (`#FF6B6B`→`error`).
- **NE PAS re-flagger** : ladders catégoriels arc-en-ciel (tags, story pickers, speakerPalette,
  `ColorGeneration`) — décision charte unique différée. Surfaces de contenu à verre derrière
  scroll (`EmojiFullPickerSheet`, `EmojiKeyboardPanel`) — anti-lisibilité HIG.

  (9 swaps `.system(size:)` → `MeeshyFont.relative`).
- **Tests** : aucun test neuf (swap mécanique ; `MeeshyFont.relative` couvert SDK + compile CI).
- **Gate** : CI `ios-tests.yml` (compile Xcode 26.1 + simulateur 18.2).
