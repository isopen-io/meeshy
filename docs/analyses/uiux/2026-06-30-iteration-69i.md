# UI/UX Analysis — Iteration 69i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **épuration de code mort** — suppression du cluster `ReplyThreadOverlay`
(ancienne implémentation de lecture de fil de réponses, **superseded** par `ThreadView`)
et de son service exclusif `ReplyThreadLoader` + son test. Itération bornée et « épurée »
dans l'esprit de 51i (suppression de `OverlayMenu`, code mort) — pas une passe mécanique
large, un seul concern : retirer un cluster mort qui pollue les audits futurs.

Vérification : la CI `ios-tests.yml` (régénère le projet via `xcodegen generate` puis
compile Xcode 26.1.x + tests simulateur 18.2) sert de build de validation — la couche
SwiftUI/UIKit ne build pas sous Linux. Le projet utilise le **globbing récursif XcodeGen**
(`project.yml`), donc supprimer les `.swift` suffit : CI régénère le `project.pbxproj` et
exclut automatiquement les fichiers supprimés (interdiction d'éditer le pbxproj à la main).

## Contexte / point de départ
Le repo contient **deux** implémentations de lecture de fil de réponses :

| Composant | Statut | Présentation | Loader | Endpoint |
|---|---|---|---|---|
| **`ThreadView`** (vivant) | rendu via `.sheet` dans `ConversationView.swift:581` (`onShowThread` → `overlayState.showReplyThread`) | feuille plein écran | `ThreadRepliesLoader` | `GET /conversations/:id/messages?replyToId=…&limit=50` |
| **`ReplyThreadOverlay`** (mort) | **jamais instancié** (vérifié repo-wide) | carte overlay flottante | `ReplyThreadLoader` | `GET /conversations/:id/threads/:parentId` |

`ReplyThreadOverlay` est l'**ancienne** implémentation (carte overlay) remplacée par
`ThreadView` (feuille plein écran). Le commentaire de `ThreadView.swift:247` la citait encore
en comparaison de comportement (« no `loadError` like ReplyThreadOverlay »), preuve de la
filiation. Son service `ReplyThreadLoader` a pour **unique** consommateur cette vue morte
(docstring : « Used by `ReplyThreadOverlay` »). `ThreadView` utilise un loader **distinct**
(`ThreadRepliesLoader`, endpoint différent) → aucun chevauchement.

## iOS Findings

### Code mort — `ReplyThreadOverlay.swift` (378 lignes, vue jamais rendue)
Vérification d'absence d'instanciation : `grep -rn "ReplyThreadOverlay(" apps/ios` → **0**
résultat hors définition. Les seules occurrences du symbole étaient des **commentaires**
(`ThreadView.swift`, `ReplyThreadLoader.swift`/`ThreadRepliesLoader.swift` docstrings) et le
`project.pbxproj` généré. Aucun `#Preview`, aucune route, aucune sheet ne la monte. Supprimée.

### Code mort — `ReplyThreadLoader.swift` (52 lignes) + son test (140 lignes)
`ReplyThreadLoader` était consommé **uniquement** par `ReplyThreadOverlay:98`
(`grep -rn "ReplyThreadLoader" apps/ios` hors fichier propre → seulement la vue morte + 2
commentaires). Une fois la vue retirée, le service est orphelin. Le garder (avec son test
`ReplyThreadLoaderTests`, 4 cas) aurait laissé un service **testé mais sans consommateur de
production** — exactement le piège de « fausse couverture » que la doctrine d'épuration
proscrit (« No temporary fixes / single source of truth »). Supprimés ensemble. `ThreadView`
n'est pas affecté : il passe par `ThreadRepliesLoader` (toujours présent + testé).

### Nettoyage des commentaires pendants
- `ThreadView.swift:247` : la comparaison « no `loadError` like ReplyThreadOverlay »
  référençait un symbole supprimé → reformulée (« no dedicated error surface (no `loadError`
  state) ») sans changer le sens.
- `ThreadRepliesLoader.swift` (docstring) : retrait des mentions « distinct from the overlay
  covered by `ReplyThreadLoader` » et « Sibling of `ReplyThreadLoader` » → docstring autonome.

## Conforme / délibérément exclu (ne pas re-flagger)
- **`ThreadData` (modèle public SDK, `packages/MeeshySDK/.../Models/ThreadModels.swift`)** :
  devient sans consommateur app après ce retrait, mais c'est un **modèle d'API public du
  SDK** (réponse du endpoint gateway `/threads/:parentId`, qui existe toujours côté backend).
  Supprimer une API **publique** du SDK est hors-scope d'une itération UI/UX iOS et plus
  risqué (consommateurs externes potentiels). **Laissé en place** — voir Différés.
- **`project.pbxproj`** : artefact généré (migration XcodeGen). **Non édité à la main**
  (doctrine `apps/ios/CLAUDE.md`). CI régénère via `xcodegen generate` → références aux
  fichiers supprimés retirées automatiquement. Le laisser périmé dans le commit est conforme.
- **`tasks/todo*.md`** : logs historiques non compilés (F1 déjà marqué « ✅ Livré »).
  Laissés tels quels (édition = bruit sans valeur de build).
- **Scrim de `ReplyThreadOverlay`** : N.B. — la surface `.ultraThinMaterial` (ligne 27) était
  un **backdrop de dimming** derrière une carte modale opaque (`surfaceGradient`), **pas** du
  chrome flottant → elle n'était de toute façon **pas** un candidat Liquid Glass (la doctrine
  51i/52i/53i réserve le verre au chrome flottant, jamais aux scrims/contenus). Point sans
  objet désormais (fichier supprimé) — noté pour clore le différé « ReplyThreadOverlay
  (.ultraThinMaterial plein écran, vérifier clip) » listé par 68i : **résolution = c'était du
  code mort, supprimé, pas converti**.
