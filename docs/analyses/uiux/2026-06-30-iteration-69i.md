# UI/UX Analysis — Iteration 69i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **accessibilité VoiceOver — boutons retour « icône-seule »**. Plusieurs en-têtes
custom rendent un bouton retour `chevron.left` **sans texte visible ni
`accessibilityLabel`**, si bien que VoiceOver annonce littéralement « chevron.left ».
Itération **bornée, additive et cohérente** (« logique épurée ») : un seul concern a11y,
une ligne par fichier, **zéro changement visuel**.

La CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de vérification
de build (pas de build local SwiftUI/UIKit sur Linux).

## Contexte / point de départ
- L'app utilise un pattern hybride NavigationStack + en-têtes custom (cf. `apps/ios/CLAUDE.md`).
  Beaucoup d'écrans masquent la barre de nav native et re-dessinent un bouton retour
  `Button { dismiss() } label: { Image(systemName: "chevron.left") … }`.
- Le composant canonique `ThemedBackButton` (`ConversationHelperViews.swift`) est **déjà**
  accessible : `.accessibilityLabel(a11y.back / a11y.back.with_unread)`.
- La clé **`a11y.back`** existe dans `Localizable.xcstrings` et est **entièrement traduite**
  (fr `Retour`, en `Back`, es `Atrás`, de `Zurück`, pt-BR `Voltar`) — c'est la SSOT pour
  ce label.
- **Important** : un `Button` icône-seule sans `accessibilityLabel` hérite du nom du symbole
  SF (« chevron.left ») comme label VoiceOver. Quand le bouton contient un **texte visible**
  « Retour » à côté du chevron, VoiceOver lit ce texte → pas de gap. Le gap n'existe que
  pour les boutons **icône-seule**.

traduite dans 5 locales**, déjà adoptée par `ThemedBackButton` → meilleur choix Prisme /

## Contexte / point de départ
- `adaptiveGlass(in:tint:interactive:)` (`packages/MeeshySDK/.../Compatibility/AdaptiveGlass.swift`)

  `CallView`, `ContextActionMenu` [51i], `MentionSuggestionPanel` + `GlobalSearchView` [68i],

- `ReplyThreadOverlay` était listé **explicitement en différé glass #1 par 68i**
  (« `.ultraThinMaterial` plein écran, vérifier clip »). C'est une **carte de contrôle
  flottante au-dessus du contenu** (thread de réponses présenté en overlay) — cas d'usage
  canonique du Liquid Glass selon la doctrine Apple (verre = chrome flottant, pas contenu).

## iOS Findings

### Adoption — carte `ReplyThreadOverlay` → Liquid Glass natif
La carte de thread utilisait `.background(RoundedRectangle(18).fill(theme.surfaceGradient(tint:))
.overlay(stroke))`. Remplacement du remplissage par
`.adaptiveGlass(in: RoundedRectangle(cornerRadius: 18, style: .continuous), tint:
Color(hex: accentColor).opacity(0.18))` **+** conservation du `.overlay(stroke)` de marque,
du `.clipShape(18)` et de l'ombre d'élévation d'accent.
- **iOS 26+** : carte en **Liquid Glass natif translucide teinté accent** — le thread flotte
  réellement en verre au-dessus de la conversation.
- **iOS < 26** : `.ultraThinMaterial` + teinte accent + hairline (via le repli de l'atome) —
  équivalent perceptuel à l'existant (`surfaceGradient` était déjà semi-translucide), plus le
  liseré qui ancre la surface.
- L'identité visuelle (accent de conversation) est **préservée** : teinte passée à l'atome +
  stroke `theme.border(tint: accentColor)` conservé au-dessus du verre. Le `.clipShape` partage
  exactement la forme du verre (RoundedRectangle 18) ⇒ pas d'artefact de clip glass-sur-bord.

### Périmètre délibérément exclu (doctrine Apple : verre = chrome, pas contenu)
- Le **scrim de fond** (`Color.black.opacity(0.3).background(.ultraThinMaterial)` plein écran,
  tap-to-dismiss) reste en material : c'est un **voile d'assombrissement de contenu**, pas du
  chrome de contrôle. Conforme, ne pas re-flagger.
- Les **fonds de lignes du thread** (`Color(hex: accentColor).opacity(0.1)` sur la bulle
  parente, chips de réponse) restent en teinte plate : contenu scrollable interne à la carte,
  pas du chrome ⇒ pas de glass-sur-glass.

### Épuration a11y — overlay de thread
1. **Bouton de fermeture sans label** : le `Button { dismiss() }` (icône `xmark.circle.fill`)
   n'exposait **aucun** `accessibilityLabel` ⇒ VoiceOver l'annonçait par le nom brut du symbole
   SF (« xmark circle fill »). Ajout de `.accessibilityLabel("common.close" → « Fermer »)`
   (réutilisation de la clé i18n existante, pas de nouvelle clé). C'est le **seul** affordance
   de fermeture exposé à VoiceOver (le scrim tap-to-dismiss et le drag-to-dismiss sont
   invisibles au rotor) — combler ce label est donc critique.
2. **Drag-indicator décoratif** (capsule de glissement) → `.accessibilityHidden(true)` : forme
   vide sans sémantique, le rotor ne doit pas s'y arrêter.
3. **Skeleton de chargement décoratif** (3 lignes shimmer) → `.accessibilityHidden(true)` :
   exact parallèle de l'épuration `mentionSkeletonRows` de 68i — avatars/barres vides sans
   contenu sémantique pendant le streaming.

## Conforme (vérifié — ne pas re-flagger)
- Le drag-to-dismiss (`DragGesture` > 100pt) et le tap-scrim de fermeture sont déjà câblés.
- Les lignes de réponse exposent déjà nom + horodatage relatif + contenu (texte natif,
  Prisme Linguistique via `translationResolver`). Conforme.
- Le `RoundedRectangle(18)` du verre == forme du `.clipShape` ⇒ pas de clip glass défectueux

### Détail module — ajout de `import MeeshyUI`
`MeeshyColors` est un **type nommé** du module `MeeshyUI` (et non une extension d'un type
du framework SwiftUI). À la différence de `Color(hex:)` — extension publique sur `Color`
qui « fuit » via la closure d'import du target — un type nommé exige un `import MeeshyUI`
explicite dans le fichier. Ajout d'`import MeeshyUI` après `import MeeshySDK` : c'est le
**pattern déjà établi** par de nombreux fichiers frères de `Components/` (`ConnectionBanner`,
`UniversalComposerBar`, `ConversationInfoSheet`, `EditPostSheet`…). `MeeshyColors` est un
`nonisolated struct` à statiques constantes — accessible sans friction depuis le contexte
MainActor de la `View`.
