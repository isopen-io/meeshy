# UI/UX Analysis — Iteration 69i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **épuration palette — alignement des indicateurs *sémantiques* de statut/confirmation
sur les tokens de marque `MeeshyColors`**. Tous les verts/oranges génériques hors-charte
(`#2ECC71`, `#F39C12`) utilisés pour exprimer un **état sémantique** (online, absent, confirmé,
prêt, actif/live) sont remplacés par les tokens `MeeshyColors.success` / `.warning` / `.info`.
Itération bornée et cohérente (« logique épurée »), dans la continuité directe du différé 68i
(« `ContactCardView` hex durs → tokens »), élargi à TOUTES les surfaces de même nature.

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur) sert de build de
validation (pas de build local SwiftUI sur Linux). Changement **pur token** (aucune logique
modifiée) ; les vues touchées sont exercées par la compilation CI + les smoke tests présence
existants (`IdentityBarElementTests`, `AvatarContextTests`).

## Contexte / point de départ
- La charte Meeshy (apps/ios/CLAUDE.md § *Semantic State Colors*) définit :
  `success = #34D399` (« Sent, **online**, **confirmed** »), `warning = #FBBF24`
  (« Attention, **pending** »), `info = #60A5FA` (« Information, help »).
- Règle SDK (`packages/MeeshySDK/CLAUDE.md`) : « Les couleurs sémantiques (error, success,
  warning) restent **statiques** via `MeeshyColors` » — jamais hardcodées.
- Plusieurs composants utilisaient encore des verts/oranges **génériques iOS** (`#2ECC71`
  vert un peu plus saturé que l'emeraude de marque, `#3498DB` bleu générique, `#F39C12`
  orange générique) pour des états qui sont **exactement** ceux couverts par la charte
  sémantique. C'est une dérive de palette : deux verts « succès » coexistaient à l'écran.

## Périmètre délibérément exclu (ladders catégoriels — décision « charte unique » différée)
Les `#2ECC71` / `#3498DB` restants dans `MeeshyUI` sont **tous** des **palettes catégorielles
arc-en-ciel** (sélecteurs multi-couleurs), pas des états sémantiques — laissés intacts,
conformément au différé 68i (« ladder catégoriel = décision charte unique, ne pas toucher au
coup par coup ») :
- `DrawingEditToolOptions.swift` (palette pinceau Story)
- `StoryComposerView.swift` / `StoryTextEditorView.swift` (palettes texte/fond Story)
- `TagInputView.swift` (ladder couleurs de tags)
- `MediaTypes.speakerPalette` (couleurs de locuteurs audio)

De même, `ColorGeneration.swift` (mapping langue/thème → couleur d'accent déterministe) et
`CoreModels`/`SampleData` (couleurs par défaut de tags/sections/attachments) sont des
**données catégorielles**, hors-scope sémantique : ne pas re-flagger.

## Conforme (vérifié — ne pas re-flagger)
- `MeeshyColors.success` (#34D399) est explicitement « Sent, online, confirmed » dans la
  charte → mapping online/confirmé/prêt/live = success est canonique.
- `MeeshyColors.warning` (#FBBF24) = « Attention, pending » → mapping away/absent = warning.
- Aucun test n'assertait ces hex (vérifié : `dotColor` est `private`, les smoke tests présence
  ne testent que la construction `.body` et les comptes structurels). Aucune baseline cassée.

## Anti-repetition check
Vérifié contre 68i (Liquid Glass champ recherche + mention ; a **différé** explicitement
`ContactCardView` hex → tokens — c'est ici qu'on solde ce différé, élargi), 52i/53i (Liquid
Glass surfaces flottantes), 51i (`ContextActionMenu`), 49i/48i (palettes SDK/app — couvraient
les **gradients de marque** indigo, **pas** les verts/oranges sémantiques). Aucune itération
antérieure n'avait aligné les indicateurs de présence/confirmation sur les tokens sémantiques.
