# UI/UX Analysis — Iteration 69i (2026-06-30)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android sont couverts par d'autres agents).
Thème : **épuration palette — couleurs sémantiques codées en dur → tokens `MeeshyColors`**
(lot 1). Itération bornée et « épurée » : on solde le différé explicite de 68i
(« `ContactCardView` hex durs `#2ECC71`/`#3498DB` → `MeeshyColors.success`/`.info` »), élargi
au seul autre usage **foreground sémantique** strictement équivalent (icône partage
`AffiliateView`). Zéro changement de comportement, zéro changement de layout.

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2) sert de
build de validation — pas de build SwiftUI local sur Linux.

## Contexte / point de départ
- La charte iOS (`apps/ios/CLAUDE.md` § Semantic State Colors + § Color Migration) impose :
  « New code MUST use the Indigo scale or semantic names ». Les **couleurs d'état
  sémantiques** ont des tokens dédiés dans le SDK (`MeeshyColors`, `MeeshyUI/Theme`) :
  `success = #34D399`, `info = #60A5FA`, `error = #F87171`, `warning = #FBBF24`.
- Plusieurs composants composaient encore des **hex bruts** quasi-identiques aux tokens
  (`#2ECC71` ≈ vert « réussi/téléphone », `#3498DB` ≈ bleu « info/email ») au lieu de
  référencer le token. Conséquences : (1) dérive de palette (deux verts différents à
  l'écran), (2) ces hex sont **figés** et n'évoluent pas si la charte ajuste le token,
  (3) ils ne passent pas par le pipeline de tokens (cohérence dark/light).
- `MeeshyColors` est **résolu app-wide** via `@_exported import MeeshyUI`
  (`apps/ios/Meeshy/MeeshyUIExports.swift`) — déjà consommé par `ContactsListTab`,
  `OnboardingStepViews`, `AffiliateView` (qui utilise déjà `MeeshyColors.error` sur le
  bouton frère « supprimer »). Aucun nouvel import requis.

## iOS Findings

### Tokenisation — `ContactCardView` (carte contact dans une bulle)
- Icône **téléphone** : `.foregroundColor(Color(hex: "2ECC71"))` → `MeeshyColors.success`.
- Icône **email** : `.foregroundColor(Color(hex: "3498DB"))` → `MeeshyColors.info`.
Les deux sont des **foregrounds sémantiques d'affordance** (vert = canal voix, bleu =
canal écrit) — exactement la sémantique `success`/`info`. Le rendu reste vert/bleu, mais
désormais aligné sur la palette de marque et tuné dark/light par le token.

### Tokenisation — `AffiliateView` (ligne de token d'affiliation)
- Icône **partager** : `.foregroundColor(Color(hex: "2ECC71"))` → `MeeshyColors.success`.
Le bouton **frère immédiat** (« supprimer ») utilise déjà `MeeshyColors.error`. La paire
action verte (partager) / action rouge (supprimer) devient cohérente : les deux passent
par les tokens, fini le vert orphelin codé en dur à côté d'un rouge tokenisé.

### Périmètre délibérément exclu (ne pas re-flagger)
- **`FeedView+Attachments.swift:1011`** — vignette placeholder de pièce jointe *location* :
  `LinearGradient([#2ECC71, #27AE60])`. C'est un **dégradé décoratif** (paire de verts ;
  `#27AE60` n'a **pas** de token correspondant), pas un foreground d'état. Tokeniser
  seulement la 1re couleur casserait l'appairage du dégradé. Laissé tel quel — conforme à
  la doctrine « tokeniser le sémantique, pas le décoratif ». À traiter, si besoin, par
  l'introduction d'un token de paire dédié, pas par swap nu.
- Les hex de **surfaces/dégradés de marque** (`#13111C`, `#09090B`, `#F8F7FF`, scales
  indigo…) et les **palettes catégorielles** (thumbnails par type) restent hors-scope :
  ce ne sont pas des couleurs d'état.

## Conforme (vérifié — ne pas re-flagger)
- `ContactCardView` : `Color(hex: accentColor)` (avatar/liséré/label) est **dynamique**
  (couleur d'accent de conversation injectée) — ce n'est PAS un hex sémantique figé, il
  reste correct. Le liséré dégradé de marque (51i/53i) et le `adaptiveGlass` (53i) sont
  conservés intacts.
- `AffiliateView` : le bouton « supprimer » utilise déjà `MeeshyColors.error`. Conforme.

## Anti-repetition check
Vérifié contre 53i (Liquid Glass `ContactCardView` — a touché le **fond** verre + liséré,
**pas** les couleurs d'icônes téléphone/email), 49i/48i (palette SDK/app — palettes
catégorielles & MeeshyUI, n'ont **pas** tokenisé ces foregrounds sémantiques de contact),
68i (qui a explicitement **listé ce point en différé**). Aucune itération antérieure
n'avait remplacé `#2ECC71`/`#3498DB` par `success`/`info`.

## Différés (pour itérations suivantes — iOS)

  token de paire) ; autres foregrounds sémantiques candidats à auditer un par un
  (`#FF6B6B`/`#4ADE80`/`#34B7F1`/`#3B82F6` — vérifier qu'ils sont bien des états et non du
  décoratif/marque-tierce avant tout swap).
- **Ladder catégoriel arc-en-ciel** (décision charte unique) : `NotificationListView.color`,
