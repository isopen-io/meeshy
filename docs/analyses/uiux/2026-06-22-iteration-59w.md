## Problème traité — accessibilité des saisies de code OTP (auth)
Les champs de saisie de **code de vérification à 6 chiffres** (OTP) sont
réimplémentés dans **4 surfaces d'authentification** distinctes, avec deux
défauts d'accessibilité :

1. **Labels a11y français figés** (rupture Prisme + a11y) — un lecteur d'écran
   anglophone/hispanophone/lusophone entendait du **français** :
   - `components/auth/recovery/OTPInput.tsx` (récupération de compte) — aucun
     hook i18n ; `aria-label="Code de vérification à 6 chiffres"` (groupe) +
     `aria-label={`Chiffre ${i} sur 6`}` (chaque chiffre).
   - `components/auth/PhoneResetFlow.tsx` (reset par téléphone) — `OTPInput`
     inline avec les **mêmes** chaînes FR figées, alors que le composant parent
     est déjà internationalisé (`useI18n('auth')`).
2. **Labels a11y absents** (lacune a11y) — les `OTPInput` inline de
   `app/auth/verify-phone/page.tsx` et `app/auth/verify-2fa/page.tsx`
   n'avaient **aucun** `role="group"` ni `aria-label` : un lecteur d'écran
   annonçait six champs texte anonymes, sans contexte « code de vérification » ni
   position « chiffre N sur 6 ».

| Surface | Avant | Après |
|---------|-------|-------|
| `recovery/OTPInput` | FR figé `Code de vérification à 6 chiffres` / `Chiffre N sur 6` | `t('otp.groupLabel', {length})` / `t('otp.digitLabel', {index,total})` |
| `PhoneResetFlow` (OTP inline) | idem FR figé | idem `t()` |
| `verify-phone` (OTP inline) | **aucun** label | `role="group"` + 2 `t()` + `autoComplete="one-time-code"` |
| `verify-2fa` (OTP inline) | **aucun** label | `role="group"` + 2 `t()` + `autoComplete="one-time-code"` |

## Décisions
- **Nouveau bloc partagé `auth.otp`** (2 clés `groupLabel`/`digitLabel`) plutôt
  qu'un label par surface : les 4 saisies OTP partagent exactement la même
  sémantique d'accessibilité. Source unique, self-documenting, parité ×4 garantie.
  Placé sous la racine `auth` (le hook `useI18n('auth')` dé-emballe la clé racine
  `auth` puis navigue `otp.*` — vérifié dans `hooks/use-i18n.ts`).
- **Paramétrage** `{length}` (groupe) et `{index}`/`{total}` (chiffre) — `verify-2fa`
  accepte une `length` variable (prop, défaut 6), les autres sont fixés à 6.
  L'interpolation `\{(\w+)\}` du `t()` est confirmée (`use-i18n.ts:186`).
- **Pas de fallback string** en 2e arg : la signature `t()` est exclusive
  (params **OU** fallback, leçon 54w). Les deux clés sont sur des surfaces
  **non visibles** (aria-label uniquement) → aucun flash de texte visible pendant
  le load async, et la parité ×4 garantit zéro clé brute affichée.
- **Bonus UX cohérence** : ajout de `autoComplete="one-time-code"` sur les saisies
  `verify-phone`/`verify-2fa` (déjà présent sur `recovery`/`PhoneReset`) — active
  l'autofill SMS natif iOS/Android sur ces deux surfaces (parité du family OTP).
- Hook `useI18n('auth')` appelé **à l'intérieur** de chaque `OTPInput` (tous des
  composants client) plutôt que prop-drilling de `t` — conventionnel et local.

## Vérifié — NE PLUS re-flagger

# Itération 59w — Analyse UI/UX (web only)

**Date** : 2026-06-22
**Périmètre** : application web (`apps/web`) exclusivement
**Veine** : a11y clavier — focus-trap des dialogues maison (complète 58w)
**Base** : `main` HEAD post-merge iter-58w `#792` (`1d1b3b6`)

## Contexte — collision 58w puis pivot

L'itération 58w (Escape + sémantique dialog sur `ConversationDrawer` et
`AgentTopicEditModal`) a été **livrée en parallèle** par #792 (`1d1b3b6`)
pendant que ce run poussait un doublon strict (#793, branche `sgqj60`). PR #793
fermée sans merge (convention #770/#783). Le run **repivote** sur le volet que
#792 a explicitement laissé en suspens : « Focus-trap (Tab cycle + focus
restore) is left out as a bounded follow-up » → c'est l'objet de 59w.

## Source de vérité

`components/v2/Dialog.tsx` (modal canonique) implémente un focus-trap inline.
Surtout, **`hooks/use-accessibility.ts` exporte déjà `useFocusTrap(containerRef,
isActive)`** — le hook a11y partagé du projet (Tab/Shift+Tab cyclent, focus le
premier élément à l'activation). Principe *Single Source of Truth* : on
**réutilise** ce hook plutôt que de réimplémenter dans chaque modale.

Constat : `useFocusTrap` n'avait **aucun consommateur** (`grep` = 0) — il
existait mais n'était câblé nulle part. 59w en fait le premier vrai usage.

## Constats

Après 58w/#792, les deux dialogues ont Escape + `role="dialog"`/`aria-modal`/
`aria-labelledby` + `aria-label` close, mais **aucune gestion du focus clavier** :
- Au Tab depuis le dernier champ, le focus **sort** de la modale vers la page
  derrière (rupture WCAG 2.4.3 — ordre de focus ; 2.1.2 — pas de piège mais le
  focus doit rester logique dans une modale).
- À la fermeture, le focus n'était **pas restauré** au déclencheur → un
  utilisateur clavier/lecteur d'écran se retrouve en haut du document.
- Lacune supplémentaire du hook lui-même : `useFocusTrap` ne **restaurait pas**
  le focus précédent au cleanup.

## Correctifs appliqués

### `hooks/use-accessibility.ts` — `useFocusTrap`
- Capture `document.activeElement` à l'activation, le **restaure** au cleanup
  (focus-restore manquant) → le focus revient au trigger à la fermeture.
- Signature élargie `RefObject<HTMLElement>` → `RefObject<HTMLElement | null>`
  (rétro-compatible ; aligne sur `useRef<HTMLDivElement>(null)` côté appelants).
- Docstring enrichie (comportement entrée/cycle/sortie).

### `components/v2/ConversationDrawer.tsx` (user-facing)
- `panelRef` (`useRef<HTMLDivElement>`) sur le panneau `role="dialog"`
- `useFocusTrap(panelRef, isOpen)` — trap actif uniquement quand ouvert ;
  restauration à la fermeture du tiroir

### `components/admin/agent/AgentTopicEditModal.tsx` (admin)
- `panelRef` sur le panneau `role="dialog"`
- `useFocusTrap(panelRef, true)` — actif tant que la modale est montée
  (le parent la monte/démonte conditionnellement)
