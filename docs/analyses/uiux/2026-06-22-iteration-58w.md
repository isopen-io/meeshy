# Analyse UI/UX — Itération 58w (web)

**Date** : 2026-06-22
**Périmètre** : application **web** uniquement (`apps/web`)
**Cible** : `components/feed/PostsFeedScreen.tsx` — écran posts du fil

## Contexte
Reprise après merge de l'iter-57wb (#780, `ReelsFeedScreen` i18n). Le cluster feed
(repéré en 53w) était presque soldé : `ReelPlayer` (57w/#774), `ReelsFeedScreen`
(57wb/#780). Restait le **plus gros volet** : `PostsFeedScreen.tsx`, l'écran posts
parité-iOS monté sur `/feed/posts` (+ alias `/feeds`).

## Constat (rupture Prisme + incohérence langue)
`PostsFeedScreen.tsx` (≈730 lignes) **n'importait aucun hook i18n**. ~30 chaînes
user-facing figées en TOUTES langues, avec en prime une **incohérence FR/EN** dans le
même fichier — symptôme classique de code écrit par à-coups :

| Catégorie | Exemples figés | Langue |
|-----------|----------------|--------|
| Toasts (succès/erreur) | `Story publiée !`, `Lien copié !`, `Reposté !`, `Cité !`, `Post supprimé`, `Erreur upload`… | **FR** |
| Chrome a11y | sr-headings (`Fil d'actualité — …`), aria sections (`Stories publiques`, `Humeurs`, `Composer une publication`), `Enregistrer un post audio`, `Mise à jour du fil…`, `Chargement de plus de publications` | **FR** |
| Temps relatif | `À l'instant`, `Il y a {n}min/h/j` (helper `formatRelativeTime`) | **FR** |
| Onglets (`FeedTabs`) | `Publications`, `Reels`, aria `Type de fil` | **FR** |
| États de chargement/erreur | `Updating...`, `Unable to load feed.`, `Retry`, `{n} new post(s)`, `No posts yet. Be the first…` | **EN** |
| Titre page | `title="Feed"` (DashboardLayout, **affiché**) | EN |

→ Un utilisateur fr voyait un mélange FR/EN ; un utilisateur en/es/pt voyait du FR
brut dans les toasts et le chrome. Double faute : Prisme (contenu/chrome non traduit)
**et** cohérence interne (deux langues dans un écran).

## Correction
- Nouveau namespace **`feed.json`** (en/fr/es/pt, ~45 clés à parité) — surface assez
  large et autonome pour justifier son propre namespace (cohérent avec `reel.json`,
  `story.json` créés pour les autres surfaces feed/deep-link).
- `useI18n('feed')` câblé dans le composant principal **et** dans `FeedTabs`
  (sous-composant exporté séparément → son propre hook).
- `formatRelativeTime(date)` → `formatRelativeTime(date, t)` (helper module-level reçoit
  `t` typé `ReturnType<typeof useI18n>['t']` — pas de `any`).
- Toasts unifiés via clés `toast.*` ; `Erreur` générique → `toast.error` réutilisé 7×.
- `Updating...` (visible) ET `Mise à jour du fil…` (sr-only live) → **une seule** clé
  `updating` (les deux disaient la même chose en deux langues).
- Pluriel `{n} new post(s)` → `newPost`/`newPosts` avec `{count}`.
- Fallbacks EN en 2e arg pour toutes les chaînes simples (anti-flash, leçon 50w).

## Validation (sandbox sans `node_modules`)
- `grep` résiduel FR/EN visible hors `mockStatuses` = **0**.
- Parité clés en↔fr↔es↔pt = OK (4×45) ; JSON valide ×4.
- `tsc` : aucune erreur propre à `PostsFeedScreen.tsx`/`feed.json` (seules erreurs =
  préexistantes « Cannot find module 'react' », deps absentes du sandbox). CI confirme le build.

## Hors périmètre (documenté — NE PAS re-flagger)
- `mockStatuses` (Marie D./Yuki T., contenus FR/JA + leur tableau `translations`) =
  **mock data** client-side (statuses pas encore câblés côté gateway, commentaire
  d'intention déjà en place l. 57-62). Sera supprimé quand l'API statuses sera branchée.
- Author fallback `'Unknown'`, `post.originalLanguage ?? 'unknown'` = données, pas chrome.
