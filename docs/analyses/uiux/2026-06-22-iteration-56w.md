# Itération 56w — Analyse UI/UX (Web)

**Date** : 2026-06-22
**Périmètre** : Application web exclusivement (`apps/web`)
**Base** : `main` HEAD post-merge 55w (#769) — commit `e1960c0`
**Thème** : Solde du cluster « micro-surfaces FR restantes » différé en 53w → 56w+ — dialogues de confirmation & flow d'enrôlement téléphone.

## Contexte
Le carry-over web (53w) listait pour 56w+ deux surfaces non internationalisées repérées dans le cluster v2/dialogues :
- `components/attachments/AttachmentDeleteDialog.tsx` (~5 chaînes de dialogue confirm FR durs)
- `components/auth/PhoneExistsModal.tsx` (~8+ chaînes FR + flow SMS)

Toutes deux affichent du **français figé en TOUTES langues** — rupture du Prisme Linguistique côté UI sur des surfaces d'action critiques (suppression de fichier, récupération/transfert de numéro à l'inscription).

## Constats

### 1. `AttachmentDeleteDialog.tsx` — 6 chaînes FR dures
| Élément | FR dur | Résolution |
|---------|--------|------------|
| `DialogTitle` | « Confirmer la suppression » | **réutilise** `attachments.contextMenu.confirmDeleteTitle` (déjà à parité 4 locales) |
| `DialogDescription` | « Êtes-vous sûr de vouloir supprimer ce fichier ? Cette action est irréversible. » | **réutilise** `confirmDeleteDescription` (la mention d'irréversibilité descend vers la note) |
| Note fichier | « Le fichier sera définitivement supprimé du serveur. » | **réutilise** `confirmDeleteIrreversible` |
| Bouton annuler | « Annuler » | `contextMenu.cancel` |
| Bouton confirmer | « Supprimer » | `contextMenu.delete` |
| État chargement | « Suppression... » | `contextMenu.deleting` |

→ **Zéro nouvelle clé** : le namespace `attachments.json` possédait déjà un bloc `contextMenu` couvrant exactement ces chaînes (FR identiques). Logique épurée : pas de doublon de clés, simple câblage `useI18n('attachments')` + fallbacks EN (leçon 50w).

### 2. `PhoneExistsModal.tsx` — ~15 chaînes FR dures
Le composant câblait déjà `useI18n('auth')` pour les **toasts** (`phoneReset.*`) mais laissait tout le **chrome visible** des 3 étapes (`choice` / `verify_code` / `success`) en FR dur :
- Étape choix : « C'est mon compte - Me connecter », « Ce n'est pas mon compte - Continuer sans numéro », « ou », « Transférer le numéro vers mon nouveau compte », « Un code SMS sera envoyé… »
- Étape vérif SMS : « Vérification SMS », « Entrez le code à 6 chiffres envoyé au {phone} », labels sr-only ×2, « Vérifier », « Renvoyer le code », « Renvoyer dans {n}s », « Annuler »
- Étape succès : « Vérification réussie ! », « Création de votre compte en cours… »

→ Nouveau sous-bloc `auth.register.wizard.phoneExistsModal` (12 clés ×4 locales) ; **réutilisation** de `phoneReset.{cancel,resendCode,success}` (FR identiques vérifiés) pour les 3 chaînes transversales — pas de duplication. Params `{phone}` / `{seconds}` interpolés via signature `t(key, params)` (cohérent avec `phoneExistsDesc` préexistant). Fallbacks EN sur les chaînes sans params.

## Vérifications
- JSON 4 locales valides + parité stricte des 12 clés `phoneExistsModal` (en/fr/es/pt).
- `register-form-wizard.test.tsx` mocke entièrement `PhoneExistsModal` → aucun impact test.
- Aucun test ne référence `AttachmentDeleteDialog`.
- Diffs locale strictement additifs (insertion ciblée après `recoverByPhone`, reste du fichier inchangé).

## Décisions d'épuration
- **AttachmentDeleteDialog** : refus d'un bloc `deleteDialog` parallèle qui aurait dupliqué les 6 chaînes déjà présentes dans `contextMenu`. Réutilisation = source unique.
- La mention « irréversible » fusionnée dans la note fichier (au lieu d'être répétée dans la description + la note) — supprime une redondance textuelle de l'UI d'origine.
