## Problème traité — rupture Prisme sur deux dialogues FR figés
Deux dialogues modaux affichaient des chaînes **françaises figées** en TOUTES
langues, contredisant le Prisme Linguistique (« le contenu traduit s'affiche
comme du contenu natif »). Le second (`PhoneExistsModal`) est une surface
d'**entrée** critique (flow d'inscription) — impact élevé.

### `components/attachments/AttachmentDeleteDialog.tsx` (aucun hook i18n)
| Chaîne FR figée | Correctif |
|-----------------|-----------|
| `Confirmer la suppression` | `t('deleteDialog.title')` |
| `Êtes-vous sûr de vouloir supprimer ce fichier ? Cette action est irréversible.` | `t('deleteDialog.description')` |
| `Le fichier sera définitivement supprimé du serveur.` | `t('deleteDialog.serverNote')` |
| `Annuler` | `t('deleteDialog.cancel')` |
| `Suppression...` / `Supprimer` | `t('deleteDialog.deleting')` / `t('deleteDialog.delete')` |

Hook `useI18n('attachments')` ajouté (le composant n'en avait aucun).

### `components/auth/PhoneExistsModal.tsx` (hook présent, JSX FR figé)
15 chaînes FR dures dans le JSX des 3 étapes (`choice`/`verify_code`/`success`)
alors que les toasts étaient déjà i18n (`phoneReset.*`). Migrées vers
`register.wizard.phoneExistsModal.*` (le composant utilise déjà `useI18n('auth')`) :
boutons `isMyAccount`/`notMyAccount`/`transferNumber`, séparateur `or`, hint SMS
`transferHint`, titre/desc `smsVerificationTitle`/`enterCodeSentTo` (param `{phone}`),
a11y `codeInputLabel`/`codeInputHint` (sr-only), `verify`/`resendCode`/`resendIn`
(param `{seconds}`)/`cancel`, succès `successTitle`/`creatingAccount`.

## Décisions
- Nouveau bloc `attachments.deleteDialog` (×4 locales, 6 clés) plutôt que de
  réutiliser `contextMenu.*` — sémantique distincte (dialogue dédié vs menu
  contextuel), self-documenting. Pas de mutualisation forcée.
- Nouveau bloc `auth.register.wizard.phoneExistsModal` (×4 locales, 15 clés) au
  même niveau que le bloc `recovery` existant.
- Fallbacks EN en 2e argument de `t()` pour les chaînes simples (anti-flash,
  leçon 50w). Les 2 clés à paramètre (`enterCodeSentTo {phone}`, `resendIn
  {seconds}`) ne prennent pas de fallback string (signature `t()` = params OU
  fallback) ; elles existent à parité ×4 locales, aucun flash de valeur.
- Parité vérifiée : `phoneExistsModal` 15 clés ×4 ; `deleteDialog` 6 clés ×4.
  JSON valide sur les 8 fichiers.
