# UI/UX Analysis — Iteration 88i (2026-07-01)

## Scope
**iOS exclusivement** (suffixe `i` — Web et Android couverts par d'autres agents).
Surface : **`DeleteAccountView`** (écran « Supprimer le compte »,
`apps/ios/Meeshy/Features/Main/Views/DeleteAccountView.swift`).
Thème : **Dynamic Type + VoiceOver** sur un **flux destructif sensible** (suppression
de compte irréversible) qui était intégralement figé en tailles de police.

Vérification : la CI `ios-tests.yml` (compile Xcode 26.1.x + tests simulateur 18.2)
sert de build de validation (SwiftUI ne compile pas sous Linux). **Aucun test neuf** :
swap de police mécanique + deux modifiers d'accessibilité déclaratifs, aucune logique
métier modifiée (parité 55i / 71i / 74i / 83i / 86i). `MeeshyFont.relative(...)` est
`public` dans `MeeshyUI` (`Theme/Accessibility.swift`), déjà importé par le fichier
(`import MeeshyUI` présent).

## Contexte / point de départ
`DeleteAccountView` (340 lignes) est l'écran de suppression de compte : carte
d'avertissement (5 pertes énumérées), champ de confirmation par phrase, bouton
destructif, puis vue de confirmation e-mail. C'est un **flux critique** — précisément
le genre d'écran où un utilisateur malvoyant (Dynamic Type XXL, VoiceOver) doit
pouvoir lire chaque conséquence avant d'agir. Deux dettes d'accessibilité y coexistaient :

1. **Dynamic Type totalement cassé** — **20** `.font(.system(size:))` figés (14/15/17
   dans le header ; 24/17/14/14/13 dans la carte d'avertissement ; 14/14/20 dans la
   confirmation ; 14/15 dans le bouton ; 64/20/15/16 dans la vue e-mail ; 12/11 dans
   le helper `sectionHeader`). Un utilisateur avec « Larger Text » ne voyait **aucun**
   agrandissement — rupture directe de la règle a11y CLAUDE.md « Use semantic fonts,
   NEVER use fixed font sizes for body text ».
2. **Navigation VoiceOver incomplète** — l'en-tête de section « Confirmation »
   (`sectionHeader`) n'exposait **pas** le trait `.isHeader` (invisible au rotor
   VoiceOver « En-têtes »), et le bloc héros de la vue de confirmation e-mail
   (icône enveloppe + titre + corps) était lu en **trois fragments** séparés au lieu
   d'un seul élément groupé.

Le reste de l'écran était déjà correct : tous les libellés sont i18n
(`String(localized:defaultValue:bundle:)`), la carte d'avertissement a déjà
`.accessibilityElement(children: .combine)`, le bouton a `.accessibilityLabel` +
`.accessibilityHint` contextuel, le titre du header a `.isHeader`, et le spacer
fantôme du header est `.accessibilityHidden(true)`.

### VoiceOver (2 améliorations idiomatiques)
1. **`sectionHeader` (helper, l.319)** — ajout de `.accessibilityAddTraits(.isHeader)` :
   l'en-tête de section « Confirmation » devient navigable au rotor VoiceOver
   « En-têtes » (cohérent avec le titre du header qui a déjà `.isHeader`, et avec
   86i qui a posé `.isHeader` sur les en-têtes de section d'`AboutView`).
2. **Bloc héros e-mail (l.266, VStack enveloppe+titre+corps)** — ajout de
   `.accessibilityElement(children: .combine)` : les trois fragments (icône décorative,
   titre « Un email de confirmation vous a été envoyé », corps explicatif) sont lus
   comme **un seul** élément VoiceOver au lieu de trois swipes (parité carte
   d'avertissement l.138 qui combine déjà).

## Différé (hors-scope 88i — signalé, NON corrigé)
- **⚠️ i18n / bug fonctionnel `requiredPhrase`** — `requiredPhrase = "SUPPRIMER MON COMPTE"`
  est une **phrase FR codée en dur** (l.19) validée côté backend via
  `AccountService.deleteAccount(confirmationPhrase:)`. Le prompt (l.160) et le
  placeholder (l.165) sont i18n'd, mais la phrase à taper reste FR quelle que soit la
  locale → un utilisateur anglophone lit « Type **SUPPRIMER MON COMPTE** » (mélange
  incohérent) et doit saisir du français. **NON corrigé ici** : c'est un changement de
  **logique** avec dépendance backend (le gateway attend probablement cette phrase
  exacte), hors du périmètre « sweep typographique + a11y, 0 logique » de cette
  itération. À traiter dans une itération dédiée avec vérification du contrat gateway.

## Vérification
- **Build** : gate = CI `ios-tests.yml` (compile Xcode 26.1.x + simu 18.2). SwiftUI ne
  compile pas sous Linux → pas de build local possible ici.
- **Tests** : aucun test neuf. Le swap `.system(size:)` → `MeeshyFont.relative(...)` est
  un changement de **présentation pur** (même `weight`/`design`, taille de base
  identique au repos, seul le scaling Dynamic Type est ajouté). Les 2 modifiers a11y
  sont déclaratifs et n'altèrent aucune logique. Couverture structurelle assurée par
  la compilation CI.
- **Régression visuelle** : nulle au réglage Dynamic Type par défaut (`.large`) — les
  text styles retournés par `textStyle(for:)` matchent les tailles d'origine à ce cran.
