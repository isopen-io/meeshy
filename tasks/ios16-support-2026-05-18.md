# Support multi-version iOS 16 / 17 / 18 — 2026-05-18

Document de référence du chantier. Toute session future de ce chantier
démarre par la lecture de ce fichier.

## Objectif

Abaisser le déploiement minimum de l'app iOS et du SDK de **iOS 17 → iOS 16**,
sans aucune régression sur iOS 17/18. iOS 16, 17 et 18 doivent rester
pleinement supportés simultanément.

## Décision de cible : iOS 16 (et non 15)

Tranché le 2026-05-18. iOS 16 livre ~95 % du gain de couverture pour ~45 % de
l'effort d'un support iOS 15 :
- iOS 15→16 ne récupère que **+1 à 2,5 %** de parc (iPhone 6s/7/SE1, A9/A10).
- iOS 15 imposerait le rework `NavigationStack`→`NavigationView` (API dépréciée
  par Apple) + fallbacks `presentationDetents` / Swift Charts / AppIntents.
- iOS 16 a déjà `NavigationStack`, détentes de sheet, Swift Charts, `ShareLink`,
  AppIntents → seul le périmètre iOS 17 reste à traiter.

## Principes directeurs (non négociables)

1. **La branche moderne EST le code actuel.** Toujours
   `if #available(iOS 17, *) { <code actuel inchangé> } else { <fallback> }`.
   Par construction, iOS 17/18 ne peut pas régresser.
2. **Centraliser, jamais disperser.** Aucun `#available` nu dans un `body`.
   Tout passe par `packages/MeeshySDK/Sources/MeeshyUI/Compatibility/`.
3. Un `Bool` calculé depuis `#available` ne débloque PAS une API pour le
   compilateur — le `#available` réel doit vivre dans le wrapper. Les flags
   `Platform.xxx` servent uniquement aux décisions de *logique*.

## Catégories d'API et stratégie

| Cat. | Type | Exemples | Traitement |
|------|------|----------|------------|
| A | Purement cosmétique | `symbolEffect` | Wrapper **no-op** sur iOS 16 |
| B | Équivalent ancien existe | `MapCameraPosition`, `containerRelativeFrame`, `scrollPosition` | Wrapper double implémentation |
| C | Aucun équivalent runtime | `@Observable` / `Observation` | Migrer vers le dénominateur commun |
| D | Feature entière indisponible | (aucune pour cible iOS 16) | — |

restent inchangés.

## Configuration à abaisser (Session 5 uniquement)

- `IPHONEOS_DEPLOYMENT_TARGET` 17.0 → 16.0 : 10× dans `project.pbxproj`, 4× dans `project.yml`
- `packages/MeeshySDK/Package.swift` : `.iOS(.v17)` → `.iOS(.v16)`

À ne faire **qu'après** encapsulation de toutes les API iOS 17, sinon build cassé.

## Journal des décisions

- **Couche `Compatibility/` dans `MeeshyUI`** — partagée par l'app et le SDK ;
  les wrappers sont des extensions `View` `public`, donc nécessitent SwiftUI.
- **`AdaptiveContentUnavailableView`** plutôt que réutiliser le composant
  existant `EmptyStateView` — afin de rendre le *vrai* `ContentUnavailableView`
  sur iOS 17+ (zéro changement de rendu sur 17/18). Discontinuité visuelle 16↔17
  acceptée (clone fidèle sur 16).
- **`symbolEffect` → no-op sur iOS 16** — effet décoratif, son absence ne change
  rien de fonctionnel.
- **`@Observable`** — décision `ObservableObject` vs `swift-perception` à
  prendre en Session 2. `swift-perception` évite la réécriture Combine du gros
  `StoryComposerViewModel` ; coût = une dépendance + wrapper `WithPerceptionTracking`.

## Protocole de vérification

Le chantier est développé en environnement Linux : **impossible de compiler/tester
un projet iOS ici** (Xcode = macOS). Entre chaque session, lancer sur macOS
depuis `apps/ios/` :

```
./meeshy.sh build && ./meeshy.sh test
```

C'est le « gate de vérification » entre sessions : ne pas démarrer la session
suivante tant que le build/test de la précédente n'est pas vert.

Note : tant que le deployment target vaut 17.0, les `#available(iOS 17.0, *)`
internes aux wrappers émettent un warning « unnecessary check » bénin. Le projet
ne traite pas les warnings comme des erreurs. Ces warnings disparaissent à la
Session 5 (bascule du target).
