# Support multi-version iOS 16 / 17 / 18 — 2026-05-18

Document de référence du chantier. Toute session future de ce chantier
démarre par la lecture de ce fichier.

## Objectif

Abaisser le déploiement minimum de l'app iOS et du SDK de **iOS 17 → iOS 16**,
sans aucune régression sur iOS 17/18. iOS 16, 17 et 18 doivent rester
pleinement supportés simultanément.

## Décision de cible : iOS 16 (et non 15)

Tranché le 2026-05-18. iOS 16 livre ~95 % du gain de couverture pour ~45 % de
l'effort d'un support iOS 15 :
- iOS 15→16 ne récupère que **+1 à 2,5 %** de parc (iPhone 6s/7/SE1, A9/A10).
- iOS 15 imposerait le rework `NavigationStack`→`NavigationView` (API dépréciée
  par Apple) + fallbacks `presentationDetents` / Swift Charts / AppIntents.
- iOS 16 a déjà `NavigationStack`, détentes de sheet, Swift Charts, `ShareLink`,
  AppIntents → seul le périmètre iOS 17 reste à traiter.

## Principes directeurs (non négociables)

1. **La branche moderne EST le code actuel.** Toujours
   `if #available(iOS 17, *) { <code actuel inchangé> } else { <fallback> }`.
   Par construction, iOS 17/18 ne peut pas régresser.
2. **Centraliser, jamais disperser.** Aucun `#available` nu dans un `body`.
   Tout passe par `packages/MeeshySDK/Sources/MeeshyUI/Compatibility/`.
3. Un `Bool` calculé depuis `#available` ne débloque PAS une API pour le
   compilateur — le `#available` réel doit vivre dans le wrapper. Les flags
   `Platform.xxx` servent uniquement aux décisions de *logique*.

## Catégories d'API et stratégie

| Cat. | Type | Exemples | Traitement |
|------|------|----------|------------|
| A | Purement cosmétique | `symbolEffect` | Wrapper **no-op** sur iOS 16 |
| B | Équivalent ancien existe | `MapCameraPosition`, `containerRelativeFrame`, `scrollPosition`, `onChange` 2-params | Wrapper double implémentation |
| C | Aucun équivalent runtime | `@Observable` / `Observation` | Migrer vers le dénominateur commun |
| D | Feature entière indisponible | (aucune pour cible iOS 16) | — |

> **Résidu majeur révélé par la revue Session 5.** L'inventaire ci-dessus,
> figé Sessions 1-4, avait omis `onChange`. La forme à deux paramètres
> `(oldValue, newValue)` est iOS 17 ; iOS 16 n'a que la forme à un paramètre
> (`onChange(of:perform:)`, dépréciée). 142 call-sites non gardés — de loin le
> plus gros bloqueur du chantier. Traité en Session 5 (voir revue).

**Non-bloqueur confirmé** : `.contentTransition(.numericText())` est disponible
dès **iOS 16.0** (seul `numericText(countsDown:)` est iOS 17). Les 12 call-sites
restent inchangés.

## Journal des décisions

- **Couche `Compatibility/` dans `MeeshyUI`** — partagée par l'app et le SDK ;
  les wrappers sont des extensions `View` `public`, donc nécessitent SwiftUI.
- **`AdaptiveContentUnavailableView`** plutôt que réutiliser le composant
  existant `EmptyStateView` — afin de rendre le *vrai* `ContentUnavailableView`
  sur iOS 17+ (zéro changement de rendu sur 17/18). Discontinuité visuelle 16↔17
  acceptée (clone fidèle sur 16).
- **`symbolEffect` → no-op sur iOS 16** — effet décoratif, son absence ne change
  rien de fonctionnel.
- **`@Observable` → `ObservableObject`** (et non `swift-perception`) — tranché
  Session 2. `ObservableObject`/`@Published`/`@StateObject` est LE pattern
  documenté du projet (app + SDK `CLAUDE.md`) ; `swift-perception` ajouterait
  une dépendance à contre-courant de la règle « dépendances minimales » du SDK.
  Aucune réécriture Combine ne s'est avérée nécessaire : les `didSet`/closures
  fonctionnent à l'identique avec `@Published`.
- **`MeeshyWidgets` reste à iOS 17.0** — tranché Session 5. L'extension widget
  s'appuie sur des API WidgetKit iOS 17 et n'a aucun bénéfice de parc à gagner
  d'un abaissement (les widgets ne sont pas un chemin critique d'adoption). Seuls
  l'app `Meeshy`, l'extension de notification, la cible de tests et `MeeshySDK`
  passent à 16.0. La couche `Compatibility/` n'a donc pas à couvrir le code widget.

## Protocole de vérification

Le chantier est développé en environnement Linux : **impossible de compiler/tester
un projet iOS ici** (Xcode = macOS). Entre chaque session, lancer sur macOS
depuis `apps/ios/` :

```
./meeshy.sh build && ./meeshy.sh test
```

C'est le « gate de vérification » entre sessions : ne pas démarrer la session
suivante tant que le build/test de la précédente n'est pas vert.

Note : tant que le deployment target vaut 17.0, les `#available(iOS 17.0, *)`
internes aux wrappers émettent un warning « unnecessary check » bénin. Le projet
ne traite pas les warnings comme des erreurs. Ces warnings disparaissent à la
Session 5 (bascule du target).

## Revue Session 2 — migration `@Observable` → `ObservableObject`

Branche `claude/ios16-session2-observable` (depuis `main`).

**App — 3 stores** (`MessageStore`, `CommentStore`, `FeedStore`) : retrait
`@Observable` + conformance `ObservableObject`, `@Published` sur les propriétés
observées (`messages`/`sections`/`unreadBelowCount`, `topLevelComments`/
`expandedThreads`, `posts`). Les `PassthroughSubject` de changement et les
observers GRDB/NotificationCenter sont conservés tels quels. Migration iso-
comportement : aucune vue SwiftUI ne dépendait du tracking Observation.

**SDK — `TimelineViewModel`** : `ObservableObject` + 13 `@Published`. `import
Observation` remplacé par `import Combine` (le fichier n'importe pas SwiftUI,
qui aurait re-exporté Combine).

**SDK — `StoryComposerViewModel`** : `ObservableObject` + 38 `@Published`. Les
6 champs d'état interne (`preloadTask` `nonisolated(unsafe)`, caches,
`_timelineViewModel`, `zIndexMap`, `nextZIndex`, `memoryObserver`) restent des
`var` simples.

**SDK — 8 vues** : `@Bindable` → `@ObservedObject`. `StoryComposerView` :
`@State` → `@StateObject` (déclaration ligne 77 **et** init repost ligne 260,
`State(wrappedValue:)` → `StateObject(wrappedValue:)` — point manqué par la
spec d'audit, rattrapé à la vérification).

**Points de vigilance pour la QA macOS** :
- Perte de granularité de tracking : `ObservableObject` notifie sur toute
  mutation `@Published` (vs tracking fin d'`@Observable`). Sur les ~38
  propriétés du composer, surveiller les re-renders du canvas/timeline.
- `$viewModel.currentSlide` (binding sur propriété calculée get/set) : valide
  avec `ObservableObject`, à confirmer au build.
- `MockStoryComposerViewModel` (tests) conforme à `StoryComposerProviding`
  uniquement — n'a pas besoin d'`ObservableObject` (aucune vue ne le lie).

### Passe de revue finale — bloqueur `onChange` (commit séparé)

La revue finale de la Session 5 a invalidé l'hypothèse « inventaire complet ».
Un `grep` ciblé sur `onChange` a révélé **142 call-sites** de
`onChange(of:initial:_:)` à deux paramètres `(oldValue, newValue)` — une API
iOS 17 — non gardés et absents de l'inventaire Sessions 1-4. Avec le target à
16.0, chacun aurait été une erreur de compilation dure.

**Créé** : `Compatibility/AdaptiveOnChange.swift` — `adaptiveOnChange(of:initial:_:)`,
extension `View` `public`. iOS 17+ : appelle `onChange(of:initial:_:)` verbatim
(zéro changement de rendu). iOS 16 : `LegacyOnChangeModifier` (`ViewModifier`)
qui mémorise la valeur précédente dans un `@State` pour reconstituer la paire
`(old, new)` par-dessus l'`onChange(of:perform:)` à un seul paramètre ; `initial:
true` déclenche aussi l'action à l'apparition avec `(value, value)`.

**Migré** : 142 sites → `.adaptiveOnChange` dans 63 fichiers (51 app, 12 SDK).
5 fichiers app (`Components/` du composer) ont reçu `import MeeshyUI`.

**Exclusions volontaires** :
- `AdaptiveMap.swift:139` — `onChange` à 2 params, mais dans le type
  `ModernInteractiveMap` déjà annoté `@available(iOS 17.0, *)` : laissé tel quel.
- 7 sites `onChange(of:) { newValue in }` à **un seul** paramètre — la forme
  iOS 14-16 (dépréciée à partir d'iOS 17). Compile sur cible 16, émet un warning
  de dépréciation bénin ; les migrer imposerait de changer la signature des
  closures, hors périmètre « bloqueurs ».
